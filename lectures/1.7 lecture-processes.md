# Лекция 7 - Процессы

Поговорим об особенностях жизненного цикла процессов в Linux: рождении, жизни и смерти. Основным предназначением операционной системы является предоставление среды для работы прикладных приложений. Бинарные файлы приложений - это всего лишь файлы, которые хранятся на диске. Для того, чтобы приложение начало свою работу, создается так называемый "процесс". 

## Определение процесса

Сложнее всего дать определение вещам, которые являются нам абстрактными, либо под одним значением определяют несколько терминов.

Так, процесс в Linux может быть:
1. Программой на стадии выполнения, а точнее **средой для работы экземпляра прогаммы**
2. Объект, которому выделено процессорное время
3. Асинхронная работа

Будем опираться на первое определение, оно лучше всего характеризует то, что нужно именно нам. Так, процесс описан некоторыми свойствами, ему выделяется часть памяти, присваевается уникальный идентификатор (PID).

Процессор обрабатывает разные процессы не одновременно, а очень быстро переключается между их выполнением. Именно так и возникает многозадачность.

Предположим, мы хотим вывести текст в терминал и используем утилиту `cat`. Конечно, мы запускаем ее, используя тот же самый терминал. При запуске будет создан процесс, ему будет выделена память, присвоен номер и так далее... А еще процесс будет дочерним, так как его породил непосредственно наш интерпретатор командной строки, чаще всего - это `bash`.

### Дерево процессов в Linux

Приложение начинает работать как только информация об экземпляре приложения заносится в таблицу процессов и планировщик начинает выделять ему процессорное время. Чтобы просмотреть все процессы воспользуемся командой `ps` с набором ключей `aux`:

```
$ ps aux
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root           1  0.0  0.1  38016 21148 ?        Ss   Sep19   0:09 /usr/lib/systemd/systemd --switched-root --system --deserialize=44 rhgb
root           2  0.0  0.0      0     0 ?        S    Sep19   0:00 [kthreadd]
root           3  0.0  0.0      0     0 ?        S    Sep19   0:00 [pool_workqueue_release]
root           4  0.0  0.0      0     0 ?        I<   Sep19   0:00 [kworker/R-rcu_gp]
root           5  0.0  0.0      0     0 ?        I<   Sep19   0:00 [kworker/R-sync_wq]
...
```

Разберем ключи:
- `a` - отобразить процессы всех пользователей.
- `u` - отобразить владельца процесса
- `x` - показать процессы без управляющего терминала (например, службы)

Вы, возможно, заметили, что в даном случае используется формат ключей, отличный от привычного. Это формат ключей BSD. При этом, при использовании привычного формата UNIX, могут возникать ошибки.

## Стадии жизни процесса

Для описания "жизненного цикла" процессов используются различные модели состояний. Для начала рассмотрим модель трех (пяти) состояний.

![Диаграмма](http://linux.opennet.ru/docs/RUS/lnx_process/p2.png)

Рассмотрим все состояния по очереди:

1. Выполнение - непосредственное выполнение процесса процессором. В этом состоянии процесс обладает всеми необходимыми ему ресурсами.
2. Ожидание - "пассивное" состояние, в котором процесс заблокирован. Он ожидает какого-то события, к примеру, получения доступа к какому-либо устройству.
3. Готовность - также "пассивное" состояние, процесс заблокирован. Однако в отличие от ожидания, заблокирован он не по своим "личным" причинам, а по команде от ОС. Дело в том, что современные ОС выполняют тысячи процессов одновременно и все они конкурируют за ресурсы: память, процессорное время. Состояние готовности означает, что процесс не ожидает никакого входного потока данных или освобождения устройства, а ожидает процессорного времени.

Из состояния готовности процесс может перейти только в состояние выолнения. В состоянии выполнения может быть ограниченное количество процессов. Для N-процессорной системы (N = количество ядер) одновременно могут выполняться только N процессов.

В любой современной ОС есть специальная программа - планировщик. Ее задача - разделять процессорное время между процессами. Процесс может перейти обратно в состояние готовности только если планировщик посчитает нужным приостановить работу процесса для перераспределения ресурсов ЦП.

Рассмотрим оставшиеся стадии. **Рождение** - пассивное состояние, при котором самого процесса еще нет, но уже готова структура для его появления. **Смерть** процесса заключается в удалении данной структуры. Кстати, бывает ситуация, когда самого процесса уже нет, а его структура все еще не удалена и висит в памяти. Такая ситуация называется зомби-процессом. Такая ситуация может произойти, например, в случае, когда процесс-потомок завершился раньше, чем того ожидал родительский процесс.

## Убить ~~билла~~ процесс

Давайте научимся убивать процессы! Для этого для начала зададимся вопросом: а как же создать процесс? Конечно, можно открыть любое прикладное приложение, но давайте сделаем это более элегантно. Запустим команду `sleep`, которая используется для создания "паузы" в выполнении скрипта. 

```
$ sleep 300d &
[1] 133769
```

Символ амперсанда в конце команды запустит ее фоном. Аргумент `300d` означает триста дней. Недурно, да? Зато точно успеем написать команду убийства. Как мы видим, при фоновом запуске, в терминал вывелся pid. Проверим, что это действительно искомый идентификатор.

```
$ ps
    PID TTY          TIME CMD
 133768 pts/0    00:00:00 zsh
 133769 pts/0    00:00:00 sleep
 133770 pts/0    00:00:00 ps
```

Действительно верный идентификатор. Теперь ликвидируем процесс командой `kill`.

```
$ kill 133769
[1]  + 133769 terminated  sleep 300d
```

## Прерывания

Прерывание — это событие, которое изменяет нормальный поток выполнения программы и может быть сгенерировано аппаратными устройствами или даже самим процессором. При возникновении прерывания текущий поток выполнения приостанавливается и запускается обработчик прерывания. После запуска обработчика прерываний предыдущий поток выполнения возобновляется. Они бывают синхронные — вызваны текущим контекстом исполнения и асинхронные — вызваны извне.

Программные прерывания являются сигналом от выполняемой программы к процессору, что требует немедленного внимания, переключая его с текущего процесса на специальный код (обработчик прерывания), который обслуживает возникшее событие и возвращает управление обратно к прерванной программе, обычно через операционную систему или системное программное обеспечение. Таким образом, программные прерывания связывают процессы с системными службами и функциями, позволяя программам запрашивать выполнение таких задач, как ввод/вывод или доступ к ресурсам, которые иначе были бы доступны только ядру операционной системы. 

Программные прерывания бывают двух основных типов:
системные вызовы и исключения. Системные вызовы — это команды, которые программа явно отправляет операционной системе для запроса каких-либо услуг (например, для работы с файлами). Исключения же возникают, когда процессор обнаруживает ошибку или нештатную ситуацию во время выполнения команды, например, деление на ноль или попытку доступа к недопустимой области памяти.

> **А ВЫ ЗНАЛИ?** OOM Killer — это
механизм ядра Linux, который предотвращает полный сбой системы при исчерпании доступной оперативной памяти (RAM), принудительно завершая один или несколько процессов для освобождения ресурсов. Он работает, вычисляя так называемый «oom_score» для каждого процесса — оценку его «плохости», основанную на потреблении памяти, наличии дочерних процессов, приоритете и других факторах. Процесс с наивысшим «oom_score» считается самым подходящим для завершения и получает сигнал SIGKILL, что позволяет системе продолжить работу.

## Разница между понятиями "Поток" и "Процесс

Поток (thread) — это, сущность операционной системы, процесс выполнения на процессоре набора инструкций, точнее говоря программного кода. Общее назначение потоков — параллельное выполнение на процессоре двух или более различных задач. Как можно догадаться, потоки были первым шагом на пути к многозадачным ОС. Планировщик ОС, руководствуясь приоритетом потока, распределяет кванты времени между разными потоками и ставит потоки на выполнение.

Процесс (process) — не что более иное, как некая абстракция, которая инкапсулирует в себе все ресурсы процесса (открытые файлы, файлы отображенные в память...) и их дескрипторы, потоки и т.д. Каждый процесс имеет как минимум один поток. Также каждый процесс имеет свое собственное виртуальное адресное пространство и контекст выполнения, а потоки одного процесса разделяют адресное пространство процесса.

## Особенности работы с клавиатурой

Если при выполнении какого-либо процесса нажать `Ctrl + C`, то это, скорее всего приведет к завершению процесса. При этом, активный процесс получит сигнал `SIGINT` (Signal Interrupt), предназначенный для прерывания процесса.
