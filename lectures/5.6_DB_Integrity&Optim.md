# Лекция 27 - Целостность и оптимизация в SQL

## Транзакции

### Блокировки

СУБД проектируются для одновременной работы множества пользователей. Когда несколько пользователей пытаются читать и изменять одни и те же данные одновременно, возникает риск нарушения целостности информации. Для решения этой проблемы используются механизмы блокировок - способы управления доступом к общим ресурсам.

Существует две стратегии блокировки:
- **С блокировкой записи и чтения**. Если пользователь запрашивает данные, тогда сервер устанавливает *блокировку чтения*, во время которой другие пользователи также могут читать эту же информацию одновременно. Если пользователь хочет изменить данные, тогда сервер устанавливает *блокировку записи*, во время которой никто другой не может получить доступ к этим данным.
- **С контролем версий (MVCC)**. Пользователи, осуществляющие запись, всё также запрашивают блокировку записи, но пользователи, запрашивающие данные, не нуждаются в блокировке. Вместо этого сервер гарантирует согласованное представление данных, то есть читатель с момента начала запроса до его завершения видит неизменное представление данных.

Также есть стратегии блокировки ресурса, которые могут выполняться на одном из трёх возможных уровнях детализации:
- **Блокировка таблицы**. Предотвращает одновременное изменение одной таблицы.
- **Блокировка страницы**. Предотвращает одновременное изменение одной страницы. Страница - минимальный объём памяти,  считываемый с диска за одну операцию. Размер страницы по умолчанию в PostgreSQL составляет 8 КБ.
- **Блокировка строки**. Предотвращает одновременное изменение одной строки.

> PostgreSQL использует блокировку с контролем версий (MVCC) и строк по умолчанию, а также предоставляет инструмент явных блокировок.

### Понятие транзакции

Транзакция - механизм группировки нескольких SQL-выражений, позволяющий успешно выполняться всем или ни одному из них. 

Представьте классический пример перевода денег с одного банковского счета на другой. Данный процесс состоит из двух операций: списание суммы с первого счёта и зачисление суммы на второй счёт. Если после выполнения первого шага произойдёт сбой, то деньги будут списаны, но не зачислены. Транзакция же гарантирует, что либо произойдут оба действия, либо не произойдёт ни одно.

### ACID

Чтобы транзакции действительно защищали данные, они должны удовлетворять четырем ключевым свойствам:
1. A - Atomicity (атомарность). Данное свойство означает, что транзакция неделима, она выполняется по принципу "всё или ничего".
2. С - Consistency (согласованность). Транзакция должна переводить базу данных из одного согласованного состояния в другое. Все правила и ограничения, определенные в БД, должны быть соблюдены.
3. I - Isolation (изоляция). Пока транзакция не завершена, её промежуточные результаты не должны быть видны другим параллельно выполняющимся транзакциям.
4. D - Durability (надёжность). После того как транзакция успешно завершена, её результаты гарантированно записываются на диск.

### Жизненный цикл транзакции

Процесс работы транзакции можно представить в виде четкой последовательности:
- Начало (`BEGIN`). Запуск транзакции, сервер фиксирует текущее состояние как точку отсчёта.
- Выполнение. Выполняются все необходимые SQL-выражения, изменения видны только внутри транзакции.
- Фиксация (`COMMIT`). Если все операции прошли успешно, то транзакция завершается командой `COMMIT`.
- Откат (`ROLLBACK`). Если в процессе выполнения возникла ошибка или было принято решение об откате, то выполняется команда `ROLLBACK`. Сервер возвращает данные к состоянию, которое было на этапе начала транзакции.

> Откат происходит не только по явной команде пользователя. Откат может быть инициирован автоматически, при аварийном завершении работы сервера. При следующем запуске сервер найдет и откатит все транзакции, которые не были зафиксированы на момент сбоя.

### Управление транзакциями

В PostreSQL реализован гибкий подход к управлению транзакциями.

#### Режим автоматической фиксации

По умолчанию (без явного начала транзакции с помощью команды `BEGIN`) включен режим автоматической фиксации (autocommit). В этом режиме каждое отдельное SQL-выражение автоматически заключается в транзакцию. Если выражение выполняется успешно, то результат сразу же фиксируется.

Режим autocommit в `psql` можно менять переменной `AUTOCOMMIT`:
- Проверить: `\echo :AUTOCOMMIT`;
- Отключить: `\set AUTOCOMMIT OFF`;
- Включить: `\set AUTOCOMMIT ON`.

> При отключении режима автоматической фиксации сессия будет находиться в транзакции постоянно, пока вы не выполните `COMMIT` или `ROLLBACK`.

#### Явное управление транзакциями

В PostgreSQL для явного начала транзакции используется команда `BEGIN`. После этого все последующие команды будут частью одной транзакции, пока вы не завершите её явно.

Для явного завершения транзакции необходимо воспользоваться `COMMIT`, которая указывает серверу пометить изменения как постоянные, если же решено не закреплять, а отменить изменения, то нужно выполнить команду `ROLLBACK`, которая указывет серверу, откатить данные до состояния, которое было до транзакции. 

Существуют сценарии, при которых транзакция может завершиться не по команде пользователя, а из-за внешних факторов, например, сбоя сервера или взаимоблокировки.

> Взаимоблокировка (deadlock) происходит, когда две транзакции ожидают ресурсы, удерживаемые другой транзакцией. При выявлении взаимоблокировки выбирается одна транзакция и осуществляется её откат, чтобы другая могла продолжить выполнение.

Рассмотрим работу транзакции на примере зачисления нового студента:
```
BEGIN;

INSERT INTO students (first_name, last_name, group_id, average_grade, birth_date)
VALUES ('Иван', 'Петров', 1, NULL, '2004-05-15')

INSERT INTO student_courses (student_id, course_id, enrollment_date)
VALUES (21, 1, '2024-09-01'),
	(21, 2, '2024-09-01'),
	(21, 3, '2024-09-01');

COMMIT;
```

Проверим результат:
```
SELECT s.student_id, s.first_name, s.last_name, c.course_name, s_c.enrollment_date
FROM students AS s
JOIN student_courses AS s_c 
	ON s.student_id = s_c.student_id
JOIN courses AS c 
	ON s_c.course_id = c.course_id
WHERE s.last_name = 'Петров' AND s.first_name = 'Иван';

 student_id | first_name | last_name |       course_name        | enrollment_date
------------+------------+-----------+--------------------------+-----------------
         21 | Иван       | Петров    | Введение в специальность | 2024-09-01
         21 | Иван       | Петров    | Математический анализ    | 2024-09-01
         21 | Иван       | Петров    | Иностранный язык         | 2024-09-01
(3 строки)
```

Видно, что студент успешно добавлен в базу и записан на три курса. Благодаря транзакции гарантируется, что не может возникнуть ситуации, когда студент есть, а записи на курсы отсутствуют, или наоборот.

### Точки сохранения транзакции

Внутри одной транзакции можно создавать промежуточные точки сохранения. Это полезно для сложных транзакций, где нужно откатить только часть операций, не отменяя всю работу целиком.

Управление точками сохранения:
- Создание точки сохранения: `SAVEPOINT my_savepoint;`
- Откат к точке сохранения: `ROLLBACK TO SAVEPOINT my_savepoint;`
- Удаление точки сохранения: `RELEASE SAVEPOINT my_savepoint;`

В прошлом примере мы добавили студента, теперь давайте добавим ему оценки по курсам. Если при добавлении одной оценки возникает ошибка, мы будем откатывать только эту оценку, сохранив предыдущие.

Добавим первую оценку:
```
BEGIN;

UPDATE student_courses 
SET grade = 5
WHERE (student_id = 21) AND (course_id = 1);

SAVEPOINT after_first_grade;
```

Добавим вторую оценку:
```
UPDATE student_courses 
SET grade = 6
WHERE (student_id = 21) AND (course_id = 2);
```

Здесь возникнет ошибка, так как в таблице `student_courses` есть проверка: `grade BETWEEN 2 AND 5`. Необходимо выполнить откат к точке сохранения:
```
ROLLBACK TO SAVEPOINT after_first_grade;
```

Добавим корректную оценку:
```
UPDATE student_courses 
SET grade = 4
WHERE (student_id = 21) AND (course_id = 2);
```

Проверим результат:
```
SELECT *
FROM student_courses
WHERE student_id = 21;

 student_id | course_id | enrollment_date | grade
------------+-----------+-----------------+-------
         21 |         3 | 2024-09-01      |
         21 |         1 | 2024-09-01      |     5
         21 |         2 | 2024-09-01      |     4
(3 строки)
```

Если же попытаться посмотреть результат в другой сессии, то мы не увидим оценок:
```
 student_id | course_id | enrollment_date | grade
------------+-----------+-----------------+-------
         21 |         2 | 2024-09-01      |
         21 |         3 | 2024-09-01      |
         21 |         1 | 2024-09-01      |
(3 строки)
```

Зафиксируем изменения:
```
COMMIT;
```

> Точка сохранения удаляется автоматически после фиксации изменения для всей транзакции.

## Индексы


При вставке строки в таблицу БД, сервер помещает её в следующую доступную ячейку памяти, поэтому таблица хранится в неупорядоченном состоянии. Поиск информации в большой таблице может занимать очень много времени. На помощь приходит инструмент, облегчающий этот поиск, - индекс.

**Индекс** - это структура данных, которая хранит содержимое выбранных столбцов в упорядоченном виде (в отличие от самой таблицы), а также ссылки (указатели) на физическое расположение соответствующих строк в таблице.

### Создание и удаление индексов

Для таблицы `students` добавим индекс, ускоряющий поиск по фамилии:
```
CREATE INDEX idx_lastname ON students (last_name);
```

После создания индекса оптимизатор запросов может использовать его, если посчитает это выгодным. Также оптимизатор выбирает, какой из индексов использовать, так как для одной таблицы их может быть несколько.

Чтобы просмотреть индексы (в PostgreSQL), воспользуемся запросом к системной таблице:
```
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'students';

   indexname   |                                   indexdef
---------------+-------------------------------------------------------------------------------
 students_pkey | CREATE UNIQUE INDEX students_pkey ON public.students USING btree (student_id)
 idx_lastname  | CREATE INDEX idx_lastname ON public.students USING btree (last_name)
(2 строки)
```

> На самом деле, уже предлагалось просмотреть информацию об индексах, когда мы вызывали описание таблицы `\d`, в этом описании также содержится информация об индексах и не только о них.

В описании индексов мы видим, что у таблицы есть ещё один индекс. Сервер БД всегда автоматически создаёт уникальный индекс для поля, являющегося первичным ключом, чтобы быстро проверять уникальность и обеспечивать быстрый доступ по этому ключу.

Если же индекс больше не нужен, его можно удалить:
```
DROP INDEX idx_lastname;
```

### Уникальные и составные индексы

Уникальный индекс, помимо ускорения поиска, также гарантирует уникальность значений в столбце. Рассмотрим таблицу `courses`. Логично, что названия курсов не должны повторяться. Создадим уникальный индекс:
```
CREATE UNIQUE INDEX idx_cname ON courses (course_name);
```

Добавление курса с уже существующим именем вызывает ошибку:
```
INSERT INTO courses (course_name, semester)
VALUES ('Базы данных', 3);

ОШИБКА:  повторяющееся значение ключа нарушает ограничение уникальности "idx_cname"
ПОДРОБНОСТИ:  Ключ "(course_name)=(Базы данных)" уже существует.
```

> Для первичного ключа (PRIMARY KEY) не нужно создавать уникальный индекс - он создаётся автоматически.

Составной индекс охватывает несколько столбцов. Например, для поиска студентов по фамилии и имени:
```
CREATE INDEX idx_fullname ON students (last_name, first_name);
```

Такой индекс полезен для поиска по имени и фамилии или только по фамилии, но бесполезен для поиска по имени, так как он упорядочивает записи сначала по фамилии, а затем уже по имени, что не даёт никакой пользы при поиске только по имени.

### Типы индексов в PostgreSQL

При выводе информации об индексах мы уже видели один из типов индексов - `btree`. PostgreSQL поддерживает несколько типов индексов для разных задач:
1. **B-tree** (Balanced-tree, сбалансированное дерево). Данный тип используется по умолчанию. Полезны для операций сравнения (в том числе `BETWEEN` и `IN`) и сортировки.
2. **Hash**. Хэш-индексы хранят 32-разрядный хэш-код, полученный из значения столбца. Следовательно, такие индексы полезны только для операции равенства `=`.
3. **GiST** (Generalized Search Tree, обобщённое поисковое дерево). Предназначены для работы с геометрическими объектами, текстами и массивами.
4. **SP-GiST** (Space-Partitioned Generalized Search Tree). Подходит для структур, в которых пространство рекурсивно разбивается на непересекающиеся области, например, префиксных деревьев.
5. **GIN** (Generalized Inverted Index, инвертированный индекс). Предназначен для массивы, JSONB, полнотекстового поиска.
6. **BRIN** (Block Range Indexes, индекс по диапазонам блоков). Используется для обработки больших таблиц, в которых значение индексируемого столбца имеет естественную сортировку.

### Использование индексов

Как узнать, использует ли сервер индекс при выполнении запроса? Для этого служит команда EXPLAIN, которая показывает план выполнения запроса, выбранный оптимизатором.

Выполним запрос и посмотрим план:
```
EXPLAIN
  	SELECT student_id
	FROM student_courses
	WHERE (student_id BETWEEN 1 AND 10) AND (course_id IN (1, 2, 5, 10));

 Seq Scan on student_courses  (cost=0.00..1.90 rows=1 width=4)
   Filter: ((student_id >= 1) AND (student_id <= 10) AND (course_id = ANY ('{1,2,5,10}'::integer[])))
(2 строки)
```

В этом случае оптимизатор выбрал полное сканирование (Seq Scan) вместо использования индекса.

Теперь создадим тестовую таблицу побольше и выполним схожий запрос:
```
EXPLAIN
	SELECT * 
	FROM test_students 
	WHERE student_id BETWEEN 1 AND 10;

                                        QUERY PLAN
-------------------------------------------------------------------------------------------
 Index Scan using test_students_pkey on test_students  (cost=0.29..20.21 rows=10 width=44)
   Index Cond: ((student_id >= 1) AND (student_id <= 10))
(2 строки)
```

Оптимизатор выбрал поиск по индексу (Index Scan).

### Достоинства и недостатки индексов

Достоинства:
- Ускоряют операции поиска, фильтрации, соединения таблиц и сортировки. Эффект наиболее заметен при работе с большими объемами данных.
- Позволяют гарантировать уникальность записей в столбце или группе столбцов (уникальные индексы).

Недостатки:
- Замедление операций записи: вставки, удаления и обновления. При каждом изменении записи в таблице приходится обновлять не только саму таблицу, но и все связанные с ней индексы.
- Занимают дополнительное место на диске, так как хранятся как отдельные структуры данных.

## Ограничения

Ограничение (Constraint) - это правило, которое накладывается на столбец в таблице БД.

Ранее уже были упомянуты некоторые ограничения, давайте рассмотрим ограничения, доступные в PostgreSQL:
1. `PRIMARY KEY` - ограничение первичного ключа. Гарантирует уникальность значений поля (или группы полей) в рамках таблицы.
2. `FOREIGN KEY` - ограничение внешнего ключа. Гарантирует, что значения поля (или группы полей) должны соответствовать значениям, содержащимся в строке другой таблицы.
3. `UNIQUE` - ограничение уникальности. Гарантирует, что данные, содержащиеся в столбце или группе столбцов, уникальны среди всех строк таблицы.
4. `CHECK` - проверочное ограничение. Ограничивает допустимые значения столбца. Позволяет указать, что значение в столбце должно удовлетворять логическому выражению.
5. `NOT NULL` - ограничение NULL-значений. Гарантирует, что поле не может принимать значение NULL.
6. `EXCLUDE` - ограничение исключения. Гарантирует, что при сравнении любых двух строк в указанных столбцах с использованием указанных операторов, хотя бы один оператор вернёт `FALSE` или `NULL`.

### Создание и удаление ограничений

Ограничения можно создавать вместе с таблицей, просто указав их в выражении `CREATE TABLE`, также их можно добавить с помощью выражения `ALTER TABLE`:
```
ALTER TABLE students 
ADD CONSTRAINT check_grade CHECK (average_grade >= 2.0 AND average_grade <= 5.0);
```

Для удаления ограничения нужно воспользоваться выражением `ALTER TABLE`, только указать не `ADD`, а `DROP`:
```
ALTER TABLE students 
DROP CONSTRAINT check_grade;
```

Для просмотра ограничений можно воспользоваться известной нам командой `\d` в psql.

### Выражения ON DELETE и ON UPDATE

При создании внешнего ключа мы сталкиваемся с проблемой: что делать, если мы пытаемся удалить запись, на которую ссылаются? Для решения этой проблемы есть выражение `ON DELETE`.

Чтобы контролировать поведение при удалении, при создании внешнего ключа указывают опцию `ON DELETE`:
- `CASCADE` - каскадное удаление связанных строк.
- `NO ACTION` (по умолчанию) - бездействие, которое часто приводит к ошибке внешнего ключа. По сути данная опция разрешает удаление и подразумевает возможность откладывания проверки ограничения внешнего ключа на более поздний этап транзакции.
- `RESTRICT` - предотвращает удаление строки, на которую ссылаются.
- `SET NULL` - установить NULL-значение.
- `SET DEFAULT` - установить значение по умолчанию. Если значение по умолчанию не удовлетворяет ограничению внешнего ключа, то операция завершится неудачей. Используется для сложных проверок, например на пересечение.

Аналогичные опции существуют для `ON UPDATE`, которые срабатывают при изменении значения связанного столбца в родительской таблице.

Рассмотрим пример каскадного удаления связей студент-курс. При удалении студента должны удаляться все записи о его курсах:
```
-- Удаляем старый внешний ключ
ALTER TABLE student_courses 
DROP CONSTRAINT student_courses_student_id_fkey;

-- Создаём новый с опцией CASCADE
ALTER TABLE student_courses 
ADD CONSTRAINT fk_sc_student 
FOREIGN KEY (student_id) 
REFERENCES students(student_id) 
ON DELETE CASCADE;

-- Проверим ограничение
 \d student_courses

...
Ограничения внешнего ключа:
    "fk_sc_student" FOREIGN KEY (student_id) REFERENCES students(student_id) ON DELETE CASCADE
...
```

Теперь при удалении студента также удалятся записи о его курсах.

## Представления

Представления (View) - виртуальная таблица, которая представляет собой сохранённый запрос. В отличие от обычной таблицы, представление не хранит данные, а хранит текст запроса и при обращении выполняет запрос, возвращая результат. Создать представление можно следующим образом:
```
CREATE VIEW students_with_groups AS
SELECT 
    s.student_id,
    s.first_name,
    s.last_name,
    g.group_name,
    s.average_grade,
    s.birth_date
FROM students AS s
LEFT JOIN groups AS g 
    ON s.group_id = g.group_id;
```

После создания представления его можно вызвать:
```
SELECT *
FROM students_with_groups
WHERE student_id BETWEEN 5 AND 15;

```

Удалить представление можно с помощью `DROP`:
```
DROP VIEW students_with_groups;
```

Представления позволяют:
- Упростить сложные запросы. Вместо того, чтобы каждый раз писать `JOIN`, можно один раз написать представление, а затем вызывать его.
- Организовать безопасность. С помощью представлений можно скрывать конфиденциальную информацию.
- Абстрагироваться от структуры данных. Можно изменить структуру таблиц, например, разделить одну таблицу на две.

## Оптимизация запросов

Мы рассмотрели четыре ключевых инструмента PostgreSQL, которые позволяют добиться надёжности и эффективности в работе с базами данных:

| Инструмент | Назначение |
|------------|------------|
| **Транзакции** | Обеспечивают атомарность операций и целостность данных |
| **Индексы** | Ускоряют поиск и фильтрацию данных |
| **Ограничения** | Гарантируют согласованность данных |
| **Представления** | Упрощают сложные запросы и повышают безопасность |

Даже с правильно спроектированной схемой данных можно столкнуться с медленной работой, если запросы написаны неэффективно. 

Основные принципы оптимизации запросов:

- Используйте индексы правильно:
  - Создавайте индексы на столбцах, часто используемых в условиях `WHERE`, `JOIN` и `ORDER BY`.
  - Не создавайте лишние индексы.
- Выбирайте только необходимые данные, не нужно выбирать все столбцы (`SELECT *`).
- Не используйте функции вычислений над индексируемыми столбцами - это делает индекс бесполезным.
- Ограничивайте количество возвращаемых строк с `LIMIT`.
- Избегайте коррелированных подзапросов.
- Используйте `EXPLAIN` для анализа плана выполнения и ищите узкие места.
