# Лекция 14 - Работа с Docker и оптимизация образов

## Основные инструкции Dockerfile

Перед тем как создавать образ с помощью Dockerfile ознакомимся с основными инструкциями. Для более подробного изучения инструкций рекомендуется обращаться к [официальной документации](https://docs.docker.com/reference/dockerfile/).

### 1. `FROM`
**Назначение**: Определяет базовый образ для сборки контейнера.  
**Синтаксис**: `FROM <image>[:<tag>] [AS <name>]`.  

### 2. `WORKDIR`
**Назначение**: Устанавливает рабочую директорию для последующих инструкций.  
**Синтаксис**: `WORKDIR <path>`.  

### 3. `COPY`
**Назначение**: Копирует файлы и директории из *source* в файловую систему образа по пути *destination*.  
**Синтаксис**: `COPY <source> <destination>`.

### 4. `ADD`
**Назначение**: Добавляет локальные или удаленные файлы и каталоги с автоматической распаковкой архивов.  
**Синтаксис**: `ADD <source> <destination>`.

### 5. `RUN`
**Назначение**: Выполняет команды во время сборки образа.  
**Синтаксис**: `RUN <command>`.

### 6. `ENV`
**Назначение**: Устанавливает переменные среды.  
**Пример**: `ENV <key>=<value>`.

### 7. `EXPOSE`
**Назначение**: Указывает, какие порты прослушивает контейнер. Это инструкция, которая не открывает порты, а служит для документирования и создания соглашения.  
**Синтаксис**: `EXPOSE <port>[/<protocol>]`.

### 8. `ENTRYPOINT`
**Назначение**: Устанавливает команду, которая **ВСЕГДА** выполняется при запуске контейнера.  
**Синтаксис**: `ENTRYPOINT ["executable", "param1", "param2"]`.

### 9. `CMD`
**Назначение**: Устанавливает команду, которая **ПО УМОЛЧАНИЮ** выполняется при запуске контейнера.  
**Синтаксис**:
- `CMD ["executable","param1","param2"]` - команда по умолчанию.
- `CMD ["param1","param2"]` - параметры по умолчанию для ENTRYPOINT.

### 10. `USER`
**Назначение**: Устанавливает пользователя и группу (опционально) для выполнения всех последующих инструкций в Dockerfile, а также для запуска контейнера.  
**Синтаксис**: `USER <user>[:<group>]`.

### 11. `HEALTHCHECK`
**Назначение**: Определяет команду для проверки "здоровья" контейнера. Docker будет периодически выполнять эту команду для мониторинга состояния контейнера.  
**Синтаксис**: `HEALTHCHECK [OPTIONS] CMD <command>`

### **Важные различия**

#### `ENTRYPOINT` и `CMD`
- **`ENTRYPOINT`** - основная команда, которая всегда выполняется.
- **`CMD`** - аргументы по умолчанию для `ENTRYPOINT` или команда по умолчанию.

#### `COPY` и `ADD`
- **`COPY`** - простое копирование файлов.
- **`ADD`** - копирование файлов с дополнительными возможностями (распаковка архивов, загрузка из URL).

## Сборка образа

Рассмотрим в качестве примера следующий Dockerfile:

```
# Используем базовый образ, копируем файлы в контейнер и устанавливаем зависимости
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

# Создаем непривилегированного пользователя
RUN useradd --create-home --shell /bin/bash appuser && \
    chown -R appuser:appuser /app
USER appuser

# Сообщаем используемый порт и определяем переменную среды
EXPOSE 5000
ENV VISITOR_NAME="Docker_Student"

# Мониторинг "здоровья" контейнера и запуск приложения
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:5000/health || exit 1
CMD ["python", "app.py"]
```

После подготовки всех необходимых для приложения файлов (исходный код, зависимости) можно приступить к сборке Docker-образа. Для этого выполним команду:

``` 
docker build -t flask-app:1.0 .
```

Флаг -t задает имя образа (flask-app) и тег версии (1.0). Тегирование крайне важно для управления разными версиями приложения.

Для просмотра списка образов используйте команду:
```
docker images
```

## Запуск контейнера и управление жизенным циклом

### Запуск контейнера:

```
docker run -P -d --name my-flask-app flask-app:1.0
```
Команда `docker run` создает и сразу же запускает контейнер из образа `flask-app:1.0` с флагами:
- `-P`: сопоставить все порты, которые контейнер объявляет как используемые (через EXPOSE в Dockerfile), со случайными свободными портами на хост-машине. Узнать, какой именно порт сопоставил Docker, можно с помощью команды: `docker port my-flask-app`.
- `-d`: Запустить контейнер в фоновом режиме.
- `--name`: Задать имя контейнеру.

В качестве альтернативы автоматическому пробросу -P существует параметр -p, предназначенный для явного связывания портов.

Дополнительные полезные флаги:  
- `-e`, `-env`: установить переменную среды.
- `--env-file`: установить переменные среды из файла.
- `-v`: монтировать volume (об этом читайте в лекции ниже).

### Управление жизненным циклом контейнера

Жизненный цикл контейнера состоит из пяти состояний:

- Создан (Created)
- Запущен (Running)
- Приостановлен (Paused)
- Остановлен (Stopped)
- Уничтожен/удален (Deleted)

![](https://media2.dev.to/dynamic/image/width=800%2Cheight=%2Cfit=scale-down%2Cgravity=auto%2Cformat=auto/https%3A%2F%2Fdev-to-uploads.s3.amazonaws.com%2Fuploads%2Farticles%2Fgevspybo00m3a7l4hfrz.png)

Основные команды для управления жизненным циклом обозначены на схеме выше. Также полезными командами являются:  
- `ps` - отобразить список контейнеров.
- `logs` - отобразить логи контейнера. При использовании флага `-f` логи контейнера выводятся в реальном времени, что удобно для мониторинга.
- `kill` - принудительное завершение работы контейнера. Полезно, если контейнер не реагирует на `stop`.

**Важные различия:**
- `pause` временно приостанавливает все процессы в контейнере, переводя его в состояние Paused, а `stop` завершает выполнение процессов, переводя контейнер в состояние Stopped.
- `kill` принудительно завершает работу контейнера, а `stop` выполняет штатное безопасное завершение работы.

### Ограничение ресурсов контейнера

Простым способом ограничения ресурсов является использования флагов команды `docker run`:

```
docker run -P -d -m 512m --cpus 2.0 flask-app:1.0
```

С помощью флага -m мы ограничиваем выделяемую память для контейнера, а с помощью флага --cpus мы ограничиваем количество ядер доступных контейнеру. Существуют и другие флаги для ограничения ресурсов контейнера, подробнее смотрите в официальной документации.

Docker Compose предоставляет декларативный и удобный способ задания ограничений на ресурсы в файле. В рамках данной лекции мы не будем углублять в Docker Compose.

### Healthcheck

Health Check - механизм, позволяющий Docker проверять, здорово ли приложение внутри контейнера и работает ли оно корректно. Данный механизм особенно полезен для оркестрации и мониторинга.

До этого мы уже реализовали HEALTHCHECK в dockerfile:
```
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:5000/health || exit 1
```
Давайте посмотрим что со здоровьем у нашего контейнера:
```
docker ps
CONTAINER ID   IMAGE           COMMAND           CREATED         STATUS                     PORTS                     NAMES
5014e05e1939   flask-app:1.0   "python app.py"   2 minutes ago   Up 2 minutes (unhealthy)   0.0.0.0:32768->5000/tcp   stoic_wozniak
```
Что-то тут не так. На самом деле проблема в том, что в базовом образе отсутствует утилита curl. Исправим это! Для этого добавим в наш Dockerfile:
```
RUN apt-get update && apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*
```
Создадим наш образ с тегом 1.1 и запустим контейнер:
```
docker build -t flask-app:1.1 .
docker run -P -d --name my-flask-app flask-app:1.1
```
Проверим здоровье нашего контейнера:
```
docker ps
CONTAINER ID   IMAGE           COMMAND           CREATED          STATUS                    PORTS                     NAMES
2399f054474a   flask-app:1.1   "python app.py"   28 seconds ago   Up 27 seconds (healthy)   0.0.0.0:32769->5000/tcp   my-flask-app
```
Отлично, теперь мы настроили HEALTHCHECK для нашего контейнера.

## Volumes

Docker Volume - это механизм сохранения и управления постоянными данными, которые должны жить дольше, чем контейнер. Volumes являются предпочтительным механизмом хранения данных, генерируемых и используемых контейнерами Docker. Volumes управляются Docker и изолированы от основных функций хост-компьютера.

### Создание и использование volume
Некоторые команды для управления volume:
```
docker volume create my_volume  # Создать volume
docker volume ls                # Просмотреть список volume
docker volume rm my_volume      # Удалить volume
```

Запуск контейнера с volume:
```
docker run -P -d --name my-flask-app -v my_volume:/app/data flask-app:1.1
```
Если volume `my_volume` не существует (как в нашем случае), то Docker создает его сам.

### Куда и как можно монтировать volume?
В примере выше показан пример монтирования volume в директорию контейнера. Кроме данного случая можно монтировать volume:  
- В файлы контейнера, например, для конфигурационного файла:  
```
docker run -v config_volume:/etc/nginx/nginx.conf nginx
```
- В несколько контейнеров одновременно:
```
docker run -P -d --name my-flask-app -v my_volume:/app/data flask-app:1.1
docker run -P -d --name my-flask-app-unhealthy -v my_volume:/app/data flask-app:1.0
```
- Read-only монтирование:
```
docker run -P -d --name my-flask-app -v my_volume:/app/data:ro flask-app:1.1
```

## Слои в контейнерах и их оптимизация

### Какие инструкции создают слои в контейнерах?

Не все инструкции в Dockerfile создают слои в финальном образе, как может показаться на первый взгляд. Рассмотрим историю сборки образа:

Можно заметить, что некоторые инструкции имеют SIZE 0B - это означает, что инструкции не вносят изменений в файловую систему, а только модифицируют метаданные.  
Инструкции, которые создают слои:  
- `RUN`
- `COPY`
- `ADD`
- `WORKDIR`, если директория создается

Инструкция `FROM` не создает слой, а начинает цепочку слоев от базового образа.

Инструкции, которые не создают слои:
- `FROM`
- `CMD`
- `ENTRYPOINT`
- `ENV`
- `EXPOSE`
- `HEALTHCHECK`
- `WORKDIR`, если директория не создается

### Оптимизация слоев

При сборке слой повторно используется из кэша, если инструкция и файлы, от которых она зависит, не изменились с момента ее предыдущей сборки. Повторное использование слоев из кэша ускоряет процесс сборки, поскольку Docker не нужно перестраивать слой заново.

#### **Техники оптимизации**

1. **Упорядочивание слоев**

Следует располагать инструкции от наименее изменяемых к наиболее изменяемым.  
Вспомним наш Dockerfile:
```
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
```
Мы могли бы сразу применить `COPY . .` и установить зависимости, однако зависимости редко изменяются, поэтому выгодно их отделить и расположить выше.

2. **Минимизация контекста сборки**

Контекст сборки - это набор файлов в директории, где находится `Dockerfile`. Наилучшим способом минимизации контекста является создание файла `.dockerignore`.  
Пример `.dockerignore`:  
```
*tmp
.git
*log
```

3. **Использование Mounts**

`Bind Mounts` Позволяют монтировать файлы с хоста непосредственно в контейнер на этапе сборки.  
Пример:
```
RUN --mount=type=bind,target=. go build -o /app/hello
```
Результат компиляции попадает в образ, но исходный код не создает отдельный слой.  
`Cache Mounts` позволяют указывать расположение кеша пакетов во время сборки.  
Пример:
```
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
```
Теперь пакеты, скачанные во время одной сборки, будут доступны при следующей сборке.

4. **Использование внешнего кеша**

По умолчанию кеш хранится локально. Каждая машина, собирающая образ будет начинать с чистого листа. Решением данной проблемы служит использование удаленного реестра для хранения кеша.  
Пример:
```
docker buildx build --cache-from type=registry,ref=user/app:buildcache .
```

## Очистка

### Команды prune

Вместо команд `rm`, которые требуют точного указания объектов для удаления, Docker предоставляет более безопасные команды `prune` для очистки неиспользуемых ресурсов.
```
docker container prune      # Удалить остановленные контейнеры
docker image prune          # Удалить висячие образы
docker volume prune         # Удалить неиспользуемые тома
docker system prune         # Удалить всё неиспользуемое (контейнеры, тома, образы, сети)
```

Вместо использования `rm` лучше использовать `prune`, потому что `prune` не может удалить работающие объекты, что делает очистку безопаснее.

Для принудительного удаления без подтверждения используйте флаг `-f`, например: `docker system prune -f`.

### Висячие образы (Dangling Images)

Типы none образов:
- Промежуточные слои сборки
- Висячие слои

Когда вы собираете образ с помощью `docker build`, Docker создает промежуточные слои. После успешной сборки эти промежуточные образы становятся висячими, так как на них больше не ссылается ни один финальный образ. Висячими слоями считаются те, которые не вошли в финальный образ.

## Подключение к оболочке контейнера и управление процессами

Для диагностики и отладки можно подключиться к оболочке работающего контейнера:
```
docker exec -it my-flask-app bash
docker exec -it my-flask-app sh
```
При подключении вы получаете интерактивную оболочку внутри контейнера.  

После подключения можно выполнять команды для диагности такие, как `env`, `ls`, `cat` и `curl`. Однако мы просто завершим работу контейнера, убив основной процесс контейнера:
```
kill 1
```

## Пользователь внутри контейнера

По умолчанию контейнеры запускаются под пользователем root, если не указана инструкция USER в Dockerfile.

### Проблемы пользователя root

1. **Угрозы безопасности**. Злоумышленник может получить доступ ко всем ресурсам выделенным контейнеру.
2. **Привилегированный доступ**. Пользователь root внутри контейнера может потенциально повлиять на хост-систему.
3. **Минимальные привилегии**. Большинству приложений не нужны права root для работы, поэтому запуск от имени root избыточен.

### Решение

Использование непривилегированного пользователя. Вспомним наш Dockerfile:
```
RUN useradd --create-home --shell /bin/bash appuser && \
    chown -R appuser:appuser /app
USER appuser
```

## Драйверы логов

Драйверы логов (Logging Drivers) - это механизмы в Docker, которые определяют куда и как отправляются логи контейнеров. Docker поддерживает различные драйверы для сбора и управления логами контейнеров.

### Примеры драйверов
1. `json-file` (по умолчанию)

```
docker run --log-driver=json-file flask-app:1.1
```

2. `journald`

```
docker run --log-driver=journald flask-app:1.1
```

3. `syslog`

```
docker run \
      --log-driver syslog --log-opt syslog-address=udp://1.2.3.4:1111 \
      alpine echo hello world
```

4. `none`

```
docker run --log-driver=none flask-app:1.1
```

## Docker squash

### Что такое `docker squash`?

`docker squash` - это инструмент (как встроенный, так и сторонний), который позволяет "сжать" несколько слоев Docker-образа в один. По сути, он создает новый образ, в котором состояние файловой системы на момент последнего слоя сохраняется, а вся промежуточная история слоев - удаляется.

### Способы использования

1. Встроенная функциональность в docker build

С помощью флага --squash:

```
docker build --squash -t flask-app:squashed .
```
Для использования флага `--squash` необходимо в конфигурации Docker daemon установить флаг experimental на `true`.

2. Сторонняя утилита `docker-squash`

Утилита работает с уже собранными образами:

```
docker-squash -t flask-app:squashed flask-app:1.1
```

`docker squash` - инструмент для уменьшения размера Docker-образа за счет объединения его слоев. Основной недостаток - отсутствие кеширования. Альтернативой является использование Multi-stage сборок.

## Multi stage сборка

### Что такое Multi-stage сборки?

Multi-stage сборки - это техника, позволяющая использовать несколько этапов (stages) сборки в одном Dockerfile. Каждый этап начинается с нового ключевого слова `FROM`, и каждый может иметь свою собственную базовую среду. Можете выборочно копировать артефакты (например, скомпилированные бинарники) из одного этапа в другой, отбрасывая всё ненужное для финального образа.

### Пример: Go-приложение

Инструменты для сборки занимают много места, и они не нужны для финального запуска готовой программы.

Dockerfile без использования multi-stage:
```
FROM golang:1.24
WORKDIR /src

COPY <<EOF ./main.go
package main
import "fmt"
func main() {
  fmt.Println("hello, world")
}
EOF

RUN go build -o /bin/hello ./main.go

CMD ["/bin/hello"]
```
Сборка образа:
```
docker build -t go-app:single-stage .
```
Dockerfile с использованием multi-stage:
```
# Сборка Builder
FROM golang:1.24 AS build
WORKDIR /src
COPY <<EOF ./main.go
package main
import "fmt"
func main() {
  fmt.Println("hello, world")
}
EOF
RUN go build -o /bin/hello ./main.go

# Финальный образ
FROM scratch
COPY --from=build /bin/hello /bin/hello
CMD ["/bin/hello"]
```
Сборка образа:
```
docker build -t go-app:multi-stage .
```
Сравним размеры образов:
```
docker images
REPOSITORY                       TAG            IMAGE ID       CREATED          SIZE
go-app                           multi-stage    6c5399995e7a   5 minutes ago    3.57MB
go-app                           single-stage   f700e8830cb3   8 minutes ago    1.36GB
```

Разница в размере получилась колоссальная, что наглядно демонстрирует полезность multi-stage сборок.

### Остановка на этапе сборки

Можно собрать только определённый этап, не доходя до финальной сборки образа. Это полезно для отладки.
```
docker build --target build -t go-app:first-stage .
```
