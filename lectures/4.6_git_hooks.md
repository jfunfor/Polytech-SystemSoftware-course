# Лекция 21 - Git hooks

Git hooks - это скрипты, которые автоматически выполняются при определенных событиях в Git. Хуки разделяются на две группы: серверные и клиентские. Хуки на стороне клиента запускаются операциями: слияние и создание коммита, а на стороне сервера они инициируются сетевыми операциями, такими как получение отправленного коммита.

Основными вариантами использования являются автоматическое тестирование и применение фильтров или проверок коммитов. Тесты являются важной частью любого рабочего процесса, и их локальный запуск поможет отловить сбои до того, как они выйдут в свет. Проверка содержимого коммитов (проверка синтаксиса, стиля кодирования и контроль формата сообщения) помогает поддерживать качество репозитория.

Хуки Git хранятся в директории `.git/hooks` репозитория. По умолчанию Git создаёт несколько примеров скриптов с суффиксом `.sample`.

## Клиентские хуки

Существует множество различных клиентских хуков:
- `pre-commit` используется для проверки для проверки данных перед созданием коммита. Запускается данный хук перед тем, как вы напечатаете сообщение коммита. Создание коммита будет отменено, если выполнение хука завершится с кодом отличным от нуля.
- `prepare-commit-msg` запускается до вызова редактора сообщения коммита, но после создания стандартного сообщения. Позволяет изменить стандартное сообщение коммита, которое открывается в текстовом редакторе при вызове команды `git commit`. 
- `commit-msg` запускается после написания сообщения коммита. Если скрипт завершается нулевым кодом, то Git отменяет создание коммита, используется для проверок сообщений коммитов.
- `post-commit` запускается после создания коммита. Обычно этот скрипт используется для уведомлений.
- `pre-push` выполняется во время выполнения команды `git push`, непосредственно до отправки данных. Его можно использовать для валидации набора изменений до их реальной отправки, нулевой код также отменяет отправку.

Рассмотрим пример `pre-commit` хука:
```
echo "Running pre-commit checks..."

CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)

if [ "$CURRENT_BRANCH" = "main" ] || [ "$CURRENT_BRANCH" = "master" ]; then
    echo "Ошибка: Нельзя коммитить напрямую в ветку $CURRENT_BRANCH"
    echo "Создайте feature-ветку: git checkout -b feature/<feature-name>"
    exit 1
fi
```

Данный хук проверяет ветку, на которой производится коммит. Если ветка является основной, то коммит отменяется:

```
$ git status
On branch main
Your branch is up to date with 'git_test/main'.

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        new file:   new_data1.json

$ git commit
Running pre-commit checks...
Ошибка: Нельзя коммитить напрямую в ветку main
Создайте feature-ветку: git checkout -b feature/<feature-name>

$ git status
On branch main
Your branch is up to date with 'git_test/main'.

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        new file:   new_data1.json

$ git checkout -b feature/add-something-new
Switched to a new branch 'feature/add-something-new'

$ git commit
Running pre-commit checks...
Commit message is valid!
[feature/add-something-new 83b9c0d] add new feature
 1 file changed, 66 insertions(+)
 create mode 100644 new_data1.json
```

## Серверные хуки

Эти скрипты выполняются до и после отправки на сервер. Pre-хуки могут возвращать ненулевой код в любой момент, что отменит передачу и отправит сообщение об ошибке клиенту.

Серверные хуки:
- `pre-recieve` запускает при старте получения данных от клиента. Если завершается с ненулевым кодом, то ни одно изменение не будет принято.
- `update` похож на `pre-recieve`, но выполняется отдельно для каждой ветки. Соответственно, если отправитель пытается обновить несколько веток, то `pre-recieve` будет вызван 1 раз, а `update` несколько. Если скрипт завершается с ненулевым кодом, то отклоняются изменения для текущей ветки, а изменения других веток могут быть применены.
- `post-recieve` выполняется после окончания процесса, может быть использован, например, для обновления сервисов или уведомления.

Создаются серверные хуки аналогично клиентским - путём добавления исполняемых файлов с определёнными именами в директорию `.git/hooks` репозитория. Cерверные хуки должны быть созданы на стороне сервера в центральном удаленном репозитории.

## Conventional Commits

Conventional Commits - спецификация для написания понятных и структурированных сообщений коммитов. Данное соглашение предоставляет простой набор правил для создания истории коммитов. Согласно спецификации, сообщение коммита должно иметь следующий формат:
```
<тип>[необязательная область изменений]: <описание>

[необязательное тело]

[необязательные сноски]
```

Сообщение коммита содержит следующие структурные элементы:
#### 1. Тип (обязательное поле) определяет цель коммита. 
Основные типы:
- `feat` - добавление функциональности.
- `fix` - исправление ошибки.
- `docs` - изменение документации.
- `style` - правки по стилю.
- `refactor` - изменения кода без исправления или добавления функционала.
- `test` - добавление или исправление тестов.
#### 2. Область изменений (необязательное поле) определяет конкретную часть проекта, которую затрагивают изменения, например: `api`, `auth`, `readme`. Указывается в круглых скобках после типа.
#### 3. Описание (обязательное поле) - краткое описание изменений в повелительном наклонении.
#### 4. Тело (необязательное поле) - более подробное описание изменений: что было сделано, почему и в чём отличия от предыдущей версии. Отделяется от описания пустой строкой.
#### 5. Сноски (необязательное поле) содержат метаинформацию, например: ссылки на задачи, информацию о критических изменениях.

Пример сообщения:
```
feat(auth): add OAuth2 authentication

- Add Google OAuth support
- Add GitHub OAuth support

Closes #123
```

Для автоматической работы с Conventional Commits можно использовать клиентские хуки:
1. `prepare-commit-msg` автоматически формирует шаблон сообщения, который подсказывает правильный формат сообщения.
2. `commit-msg` проверяет соответствия формату, может автоматически исправлять мелкие ошибки, например регистр.

Также существуют инструменты для работы с Conventional Commits: Commitizen - интерактивный помощник для коммитов, который предоставляет интерфейс для создания коммитов и автоматически генерирует сообщения в правильном формате, Husky - менеджер Git Hooks, который упрощает настройку хуков.
