# Лекция 10 - Протоколы прикладного уровня - SSH, TLS, DNS, HTTP и gRPC

Протоколы прикладного уровня обеспечивают взаимодействие сети и пользователя. Прикладной уровень разрешает приложениям пользователя иметь доступ к сетевым службам, таким как обработчик запросов к базам данных, доступ к файлам, пересылке электронной почты. Рассмотрим основные протоколы прикладного уровня.

## SSH

SSH (Secure Shell) — это сетевой протокол, который позволяет устанавливать безопасное соединение с удаленным компьютером. SSH шифрует все данные, передаваемые между клиентом и сервером, что защищает от прослушивания и перехвата. SSH работает по умолчанию на порту 22.

SSH работает по классической клиент-серверной архитектуре:

- SSH-сервер (`sshd`) - это программа-демон, которая постоянно работает на удаленной машине, ожидая входящих подключений.
- SSH-клиент (`ssh`) - это программа, которая используется для подключения к серверу.

### Аутентификация SSH

SSH-сервер поддерживает 3 способа аутентификации:

- По IP-адресу. Предполагается, что все подключения с указанного адреса автоматически принимаются. Этот способ небезопасен и используется крайне редко.
- По логину и паролю. Наиболее привычна для пользователей, но пароль можно перебрать или перехватить.
- По ключу. Является наиболее безопасным вариантом.

#### Аутентификация по ключу

Для каждого пользователя генерируется своя уникальная пара ключей: закрытый ключ хранится у пользователя, а открытый размещается на сервере.

Имея закрытый ключ, можно по специальному алгоритму восстановить открытый ключ. Но обратное невозможно - по открытому ключу вычислить закрытый ключ нельзя, можно только проверить соответствие. По открытому ключу сообщения зашифровываются, а по закрытому - расшифровываются. Таким образом, только владелец ключа может расшифровать предназначенное ему сообщение.

Для генерации ключей предназначена утилита `ssh-keygen`. Генерация SSH-ключа производится следующим образом:
```
$ ssh-keygen -t <type>
```

Параметр `–t` задает тип `<type>` создаваемого ключа.

После выполнения команды `ssh-keygen` будут созданы два файла - с закрытым и открытым ключом. Публичную часть SSH-ключа (содержимое файла с расширением .pub) нужно поместить на сервер.

Скопировать публичный ключ на удаленный хост можно с помощью команды `ssh-copy-id`:
```
$ ssh-copy-id <пользователь>@<IP-адрес>
```

Подключение к удаленному хосту:
```
$ ssh <пользователь>@<IP-адрес>
```

После подключения IP-адрес добавляется в файлы known_hosts и config на компьютере. Публичные ключи пользователей хранятся в файле `~/.ssh/authorized_keys` в домашнем каталоге пользователя на удаленном хосте.

### Конфигурационный файл

Конфигурационный файл SSH (`~/.ssh/config`) - файл на стороне клиента, который позволяет задавать параметры для подключения к удаленным хостам. Благодаря конфигурационному файлу нет необходимости каждый раз в командной строке прописывать параметры подключения. Пример конфигурационного файла:
```
Host myhost
    HostName 192.168.1.100
    User myuser
    IdentityFile ~/.ssh/mykey
```

После описания config-файла можно подключаться к удаленному хосту с помощью команды:
```
$ ssh myhost
```

### Jump host

Jump host (или bastion host) - это промежуточный сервер, который используется для доступа к другим серверам во внутренней сети, которые не имеют прямого доступа к интернету. Jump host используется для повышения безопасности, так как позволяет ограничить доступ к внутренним серверам только через один защищенный хост.

Для подключения к целевому серверу необходимо добавить опцию `-J`:
```
$ ssh -J jumpuser@jumpserver:port finaluser@finalserver
```

Также можно настроить config:
```
Host jumpserver
    HostName jump.example.com
    User username
    IdentityFile ~/.ssh/jump_key

Host internal-server
    HostName 10.0.1.100
    User admin
    ProxyJump jumpserver
```

### Передача файлов

SCP (Secure Copy Protocol) - утилита для безопасной передачи файлов через SSH соединение.

Копирование файлов на удаленный хост:
```
$ scp <файл> <пользователь>@<IP-адрес>:<путь_назначения>
```

Копирование файлов с удаленного хоста:
```
$ scp <пользователь>@<IP-адрес>:<путь_к_файлу> <путь_назначения>
```

Рекурсивное копирование:
```
$ scp -r <каталог> <пользователь>@<IP-адрес>:<путь_назначения>
```

### Zsh и автодополнение SSH

Zsh - командная оболочка, которая имеет множество полезных функций, включая автодополнение. Суть автодополнения заключается в том, что `zsh` автоматически читает файл `~/.ssh/config` и использует его для дополнения. Допустим у нас имеется конфигурационный файл:
```
Host myhost
    HostName 192.168.1.100
    User myuser
    IdentityFile ~/.ssh/mykey
```

Теперь при вводе `ssh my` автоматически дополнится до `ssh myhost`.

## TLS

TLS (Transport Layer Security) - криптографические протокол, обеспечивающий защищённую передачу данных между узлами в сети Интернет.

Основные функции TLS:

- **Шифрование**. Скрывает данные, передаваемые от третьих лиц.
- **Аутентификация**. Позволяет убедиться, что стороны, обменивающиеся информацией, действительно те, за кого они себя выдают.
- **Целостность**. Гарантирует, что данные не были потеряны, повреждены, подделаны или сфальсифицированы.

### TLS и SSL

TLS - это прямой преемник и более новая, безопасная версия протокола SSL. Основные отличия SSL и TLS можно представить в сравнительной таблице:

| Характеристика        | SSL                                     | TLS                                          |
|-----------------------|-----------------------------------------|----------------------------------------------|
| Статус                | Устаревший (все версии)                 | Активно используется (TLS 1.2, 1.3)          |
| Последняя версия      | SSL 3.0 (1996 г.)                       | TLS 1.3 (2018 г.)                            |
| Уровень безопасности  | Низкий, содержит множество уязвимостей  | Высокий, особенно в версиях 1.2 и 1.3        |
| Поддержка в браузерах | Отключена во всех современных браузерах | Полная поддержка, приоритетное использование |
| Совместимость         | Ограниченная                            | Широкая, с обратной совместимостью           |

### Сравнение версий TLS

Версии TLS

- TLS 1.0: Введен в 1999 году как более безопасная альтернатива SSL 3.0, но сейчас считается устаревшим.
- TLS 1.1: Введен в 2006 году, добавляет защиту от некоторых типов атак, таких как атаки на блочное шифрование (CBC).
- TLS 1.2: Введен в 2008 году, широко используется, поддерживает современные криптографические алгоритмы и все еще актуален.
- TLS 1.3: Введен в 2018 году, оптимизирован для повышения производительности и безопасности, сокращает количество раундов, необходимых для установления защищенного соединения.

Сравнительная таблица версий TLS:

| Характеристика                       | TLS 1.0      | TLS 1.1    | TLS 1.2    | TLS 1.3          |
|--------------------------------------|--------------|------------|------------|------------------|
| **Год выпуска**                      | 1999         | 2006       | 2008       | 2018             |
| **Статус**                           | Устаревший   | Устаревший | Актуальный | Рекомендуемый    |
| **Безопасность**                     | Низкая       | Средняя    | Высокая    | Максимальная     |
| **Производительность**               | Низкая       | Средняя    | Высокая    | Оптимизированная |
| **Поддержка современных алгоритмов** | Ограниченная | Частичная  | Полная     | Улучшенная       |

### Шифрование

Протокол TLS для обеспечения безопасности использует гибридную систему шифрования, которая сочетает в себе преимущества как асимметричного, так и симметричного шифрования.

Симметричное шифрование предполагает наличие одного секретного ключа. Применение одного ключа для шифрования и дешифрования делает данный метод простым. Возникает проблема передачи секретного ключа по незащищенному каналу. Популярные алгоритмы симметричного шифрования: AES, ChaCha20.

Асимметричное шифрование предполагает наличие пары ключей - открытого и закрытого. Открытый ключ свободно распространяется и включается в сертификат сервера. Любой клиент может использовать его для шифрования данных, которые сможет расшифровать только владелец закрытого ключа. Закрытый ключ должен храниться в секрете на сервере. Используется для расшифровки данных, зашифрованных открытым ключом, и для создания цифровой подписи. Популярные алгоритмы асимметричного шифрования: RSA, ECDSA.

Использование обоих типов шифрования помогает протоколу TLS держать баланс между скоростью работы и безопасностью. Асимметричное шифрование используется на начальном этапе соединения для обмена ключами и аутентификации, а симметричное - для шифрования непосредственно сообщений.

Для работы асимметричного шифрования необходимо сгенерировать пару ключей. Рассмотрим пример генерации с использованием утилиты `OpenSSL`.

Генерация приватного ключа:
```
$ openssl genpkey -algorithm RSA -out private.key -pkeyopt rsa_keygen_bits:2048
```

Извлечение публичного ключа из сгенерированного приватного:
```
$ openssl rsa -pubout -in private.key -out public.key
```

### Аутентификация

Аутентификация в TLS обеспечивает проверку подлинности сторон через сертификаты доверенных центров (Certificate Authority, CA). Сертификат может запросить как клиент, так и сервер. Это защищает соединение от MITM-атак (Man-in-the-Middle - человек посередине), когда третья сторона вмешивается в соединение двух участников и перехватывает закрытые ключи.

### Целостность данных

Для обеспечения целостности данных в TLS используется хеширование. Передаваемые данные дополняются криптографической контрольной суммой, получатель вычисляет хеш заново и сравнивает его с полученным. Совпадение хеш-сумм позволяет убедиться в том, что информация пришла в исходном виде.

### Установление безопасного соединения (TLS Handshake)

TLS Handshake - процесс, который запускает сеанс связи с использованием TLS. Во время рукопожатия клиент и сервер обмениваются сообщениями, чтобы подтвердить друг друга, установить алгоритмы шифрования и согласовать ключи сеанса. 

Основные этапы:

- **Согласование параметров (Client & Server Hello)**. Клиент и сервер согласуют версии TLS и алгоритм шифрования, который они оба могут использовать.
- **Аутентификация**. Сервер (при необходимости и клиент) отправляет клиенту свой цифровой сертификат. Клиент проверяет его и убеждается в подлинности сервера.
- **Обмен ключами (Key Exchange)**. Безопасная генерация общего секретного сеансового ключа для симметричного шифрования.

Рассмотрим обмен ключами двумя различными алгоритмами:

- RSA: Клиент генерирует ключ и шифрует его открытым ключом сервера. Расшифровать может только владелец закрытого ключа.
- ECDHE: Клиент и сервер обмениваются открытыми параметрами и независимо вычисляют одинаковый секрет. Этот метод обеспечивает совершенную прямую секретность (PFS), защищая даже прошлые сеансы.

После генерации одинаковых сеансовых ключей дальнейшее общение происходит с использованием быстрого симметричного шифрования.

### Цифровой сертификат и его жизненный цикл

Цифровой сертификат - это электронный документ, который удостоверяет принадлежность открытого ключа определенному субъекту.

Процесс аутентификации в TLS основан на модели доверия, ключевым элементом которой является удостоверяющий центр (Certificate Authority, CA). Это доверенная третья сторона, которая выпускает цифровые сертификаты. 

Жизненный цикл цифрового сертификата:

1. Генерация. Создается пара ключей и CSR (Запрос на подписание сертификата).
2. Проверка. CA проверяет данные владельца.
3. Выпуск. СА подписывает CSR и выдает сертификат.
4. Использование. Сертификат устанавливается на сервер.
5. Продление. Процесс повторяется до истечения срока действия.
6. Отзыв. При компрометации ключа сертификат досрочно вносится в CRL (Certificate Revocation List, список отозванных сертификатов).

Пример генерации самоподписанного сертификата:
```
$ openssl req -x509 -nodes -days 365 -newkey rsa:2048
```

### Let's Encrypt

Let's Encrypt - это бесплатный, автоматизированный и открытый центр сертификации (Certificate Authority, CA), который был создан для того, чтобы сделать защищённые HTTPS-соединения повсеместными в интернете.

В основе работы Let's Encrypt лежит протокол ACME (Automated Certificate Management Environment). Этот протокол определяет, как клиентское программное обеспечение может автоматически взаимодействовать с центром сертификации для выполнения операций:

- Верификация домена. Let's Encrypt должна убедиться, что вы контролируете домен, для которого запрашиваете сертификат.
- Выпуск сертификата. После успешной проверки клиент получает подписанный сертификат.
- Автопродление. ACME-клиент может быть настроен на автоматическое продление сертификатов до истечения их срока действия.

Пример команды для получения сертификата с использованием `certbot`:
```
$ certbot --nginx
```

## DNS

DNS (Domain Name System) - это система, которая переводит удобочитаемые доменные имена в IP-адреса, необходимые компьютерам для связи. DNS использует порт 53 по протоколам UDP и TCP. UDP является основным и наиболее распространенным протоколом для DNS-запросов.

### FQDN

FQDN (Fully Qualified Domain Name) - полное доменное имя, которое однозначно идентифицирует конкретный ресурс в глобальной сети. Пример: `www.google.com.`.

Состав FQDN:

- `www` - имя хоста (третий уровень).
- `google` - домен второго уровня.
- `com` - домен верхнего уровня (TLD).
- `.` - корневой уровень (обычно пропускается в браузерах).

TLD - самый высокий уровень в иерархии DNS после корневого. TLD делятся на основные категории:

- Административные (gTLD). Указывают тип организации, обычно состоит из трех букв. Пример: `.com`, `.org`, `edu`.
- Географические (ccTLD). Указывает принадлежность домена к той или иной территории. Пример: `.ru`, `kz`, `de`.

### DNS зона

DNS-зона - это административная единица, часть доменного пространства, которой управляет конкретная организация или администратор. Домен может быть разделен на множество зон, что делает систему распределённой и гибкой в управлении.

Иерархия DNS зон соответствует структуре доменного имени:

- Корневая зона (Root zone) - высший уровень системы DNS, обозначается точкой (`.`). Поддерживается корневыми серверами, управляется организацией IANA.
- Зоны доменов верхнего уровня (TLD zones) - управляют доменами верхнего уровня (TLD). За разные TLD отвечают разные организации.
- Зоны доменов второго уровня (Second level zones) - то, с чем чаще всего имеют дело владельцы сайтов. Владельцы домена управляют этой зоной, они могут создавать поддомены.
- Зоны обратного просмотра (Reverse DNS Zones) - специальные зоны, используемые для преобразования IP-адресов в доменное имя. Это важно для проверки подлинности сервера.

Каждая DNS-зона содержит ресурсные записи — инструкции, которые описывают, как обрабатывать запросы для данного домена. 

Основные типы записей:

- A (address record) - связывает доменное имя с IPv4-адресом.
- AAAA (Quad-A record) - связывает доменное имя с IPv6-адресом.
- CNAME (Canonical Name) - создает псевдоним, перенаправляя одно доменное имя на другое.
- MX (Mail Exchange) - указывает на почтовые серверы для этого домена.
- TXT (Text record) - содержит текстовую информацию, используется для проверок владения доменом.

### Wildcard

Wildcard = это специальный символ `*`, используемый в DNS-записях для создания универсального правила, которое соответствует любому поддомену на указанном уровне. Таким образом, запись `*.example.com` отвечает на запросы: `shop.example.com`, `blog.example.com`, но не отвечает на запись `dev.test.example.com`, т. к. `*` покрывает только один уровень.

### Принцип работы DNS

Процесс начинается с отправки запроса на ближайший DNS-сервер, который обычно предоставляется интернет-провайдером. Данный сервер называется рекурсивным резолвером. Если этот сервер знает нужный IP-адрес, то он его возвращает, если нет, то запрос передается на другие DNS-серверы.

Процесс поиска IP-адреса:

1. Локальный кеш. Сначала браузер проверяет свой локальный кеш на наличие записи для доменного имени.
2. Рекурсивный резолвер. Если записи нет в кеше, то отправляется запрос на рекурсивный DNS-сервер. Этот сервер выполняет роль посредника, который ищет нужную информацию.
3. Корневые серверы. Если рекурсивный резолвер не знает IP-адрес, то он отправляет запрос на корневой сервер. Корневые серверы знают, какой сервер отвечает за каждый домен верхнего уровня (TLD), например, `.com`. Корневой сервер отвечает, какой сервер управляет доменом `.com`, рекурсивный резолвер отправляет запрос на этот сервер.
4. Сервер TLD. Сервер TLD отвечает, какой сервер управляет требуемым доменом. Рекурсивный резолвер направляет запрос на авторитетный DNS-сервер, который содержит точные данные.
5. Авторитетный сервер. Авторитетный сервер возвращает IP-адрес, связанный с разыскиваемым доменом, рекурсивному резолверу, который отправляет IP-адрес браузеру.

### Инструменты для работы с DNS

#### 1. `nslookup` (Name Server Lookup) -  это инструмент командной строки, используемый для запроса DNS-серверов с целью получения информации о доменных именах, IP-адресах и других DNS-записях.
 
Запрос А-записи:
```
$ nslookup -type=A yandex.ru

Server:         10.255.255.254
Address:        10.255.255.254:53

Non-authoritative answer:
Name:   yandex.ru
Address: 77.88.55.88
Name:   yandex.ru
Address: 77.88.44.55
Name:   yandex.ru

```

Запрос записи типа MX:
```
$ nslookup -type=MX yandex.ru

Server:         10.255.255.254
Address:        10.255.255.254:53

Non-authoritative answer:

```

Обратный запрос:
```
$ nslookup 77.88.55.88

Server:         10.255.255.254
Address:        10.255.255.254:53

Non-authoritative answer:

```

#### 2. `dig` (Domain Information Groper) - более мощный и современный инструмент.

Примеры использования:
```
dig example.com A       # Запрос записи типа А
dig example.com TXT     # Запрос записи типа TXT
dig example.com +trace  # Отслеживание цепочки DNS-запросов
``` 

#### 3. `host` - это простая утилита для выполнения DNS-запросов.

Пример использования:
```
host -t TXT example.com

example.com descriptive text "v=spf1 include:_spf.example.com ~all"
```

## HTTP

HTTP (HyperText Transfer Protocol) - протокол передачи гипертекста в формате HTML. Как протокол передачи данных, HTTP определяет структурные правила взаимодействия между клиентом и сервером. Чаще всего с помощью HTTP передают веб-страницы, то есть контент сайтов, которые отображаются в интернете.

### Идентификация ресурсов

URI (Uniform Resource Identifier) - унифицированный идентификатор ресурса, представляющий собой строку символов для однозначной идентификации абстрактного или физического ресурса.

URL (Uniform Resource Locator) - подмножество URI, которое не только идентифицирует ресурс, но и предоставляет информацию о его местоположении и методе доступа. Используется для веб-навигации, API.

Компоненты URL:

- Схема (протокол) - схема обращения к ресурсу,например, http, https, ftp.
- Авторизация - логин и пароль, например, user:password@.
- Хост - доменное имя или IP-адрес.
- Порт - порт хоста для подключения.
- Путь - уточняющая информация о местонахождении ресурса.
- Запрос - строка запроса с передаваемыми на сервер (методом GET) параметрами. Начинается с символа `?`, разделитель параметров — знак `&`.
- Фрагмент - необязательный компонент, указывающий на конкретное место (якорь) на странице. Фрагменту предшествует символ `#`.

URN (Uniform Resource Name) - подмножество URI, обеспечивающее постоянное уникальное имя ресурса независимо от его местоположения. Используется для идентификации книг, RFC-документов.

### Версии HTTP

1. HTTP/0.9 (1991). Протокол поддерживал только запросы типа GET, не существовало заголовков, статусов или других методов.
2. HTTP/1.0 (1996). Появились заголовки, коды состояний, поддержка других типов данных и новые методы (POST, HEAD).
3. HTTP/1.1 (1999). Введен режим "постоянного соединения" (TCP-соединение может оставаться открытым после отправки ответа на запрос), обязательный заголовок Host. Появились новые методы: PUT, DELETE, OPTIONS, TRACE.
4. HTTP/2.0 (2015). HTTP/2 в отличие от предыдущих версий является бинарным. Появилось мультиплексирование, которое позволяет отправлять несколько запросов и получать ответы одновременно через одно соединение TCP. Также появились: приоритизация запросов, сжатие заголовков, server push (сервер сам отправляет ресурсы, которые понадобятся).
5. HTTP/3.0 (2019). Использует UDP (QUIC) вместо TCP. Устранена проблема "блокировки заголовков", при которой потеря одного пакета может задержать загрузку всей веб-страницы.

### Структура HTTP-сообщения

HTTP-сообщение состоит из трех основных компонентов:

- Стартовая строка, в которой определяется адрес, по которому отправляется запрос, и тип сообщения. Указывается метод, который определяет действия при получении этого сообщения.
- Заголовки (не обязательно) характеризуют тело сообщения, параметры передачи и прочие сведения.
- Тело сообщения (не обязательно) - непосредственно данные сообщения. Обязательно должно отделяться от заголовков пустой строкой.

Пример HTTP-запроса:
```
POST /api/users HTTP/1.1                            # Стартовая строка
Host: api.example.com                               # Заголовки
Content-Type: application/json
Content-Length: 56

{"name": "John Doe", "email": "john@example.com"}   # Тело сообщения
```

### Основные методы HTTP

1. **GET**. Используется для запроса данных с сервера. Позволяет получить, но не изменять данные.
2. **POST**. Используется для отправки данных на сервер для создания или обновления запрашиваемого ресурса. Данные для этого указываются в теле запроса.
3. **PUT**. Также используется для отправки данных на сервер для создания или обновления запрашиваемого ресурса. В отличие от POST не требует от сервера ответа.
4. **PATCH**. Используется для частичного изменения ресурса. Включает в себя инструкцию по обновлению, а не полностью новое содержимое, чем и отличается от метода PUT.
5. **DELETE**. Полностью удаляет информацию о запрашиваемом ресурсе.
6. **HEAD**. Идентичен GET, но без тела ответа. Используется для получения мета-информации, проверки наличия ресурса и чтобы узнать, не изменился ли ресурс с момента последнего обращения.
7. **OPTIONS**. Запрос поддерживаемых методов HTTP для ресурса и других параметров коммуникации. Используется для определения возможностей веб-сервера или параметров соединения для конкретного ресурса. 

Безопасные методы - методы, которые не изменяют состояние сервера, выполняют операции read-only. К безопасным методам относятся: GET, HEAD и OPTIONS.

Идемпотентные методы - методы, которые при многократном выполнении дают идентичный результат. К ним относятся: GET, HEAD, OPTIONS, PUT, DELETE.

### HTTP-статусы

HTTP-статусы - это стандартизированные трёхзначные числа, которые веб-сервер передаёт в ответ на запрос клиента. Они показывают, как именно сервер обработал запрос: успешно или с ошибкой.

Классы статусов:

- 1xx - Информационные. Указывают, что запрос был получен и процесс его обработки продолжается.
- 2xx - Успех. Указывают на успешную обработку запроса.
- 3xx - Перенаправление. Указывают, что клиент должен выполнить дополнительные действия для завершения запроса.
- 4xx - Ошибка клиента. Означают, что запрос содержит ошибку или не может быть выполнен.
- 5xx - Ошибка сервера. Указывают, что сервер не смог корректно обработать запрос.

### Управление состоянием и доступом

HTTP позволяет не только получать данные, но и управлять взаимодействием между клиентом и сервером. Разберем основные механизмы этого управления.

#### 1. Управление кешированием

Кеширование позволяет хранить копии ресурсов на стороне клиента, чтобы уменьшить нагрузку на сервер и ускорить загрузку страниц. Управление кешированием производится с помощью следующих заголовков:

- Заголовок `Cache-Control` управляет кешированием с помощью директив, определяющих кешируемость ответа, время жизни кеша, требования валидации.
- Заголовок `Expires` указывает дату и время, когда кеш становится невалидным. Использует абсолютное время из-за чего может произойти рассинхронизация, в `Cache-Control` используется относительное время (с момента получения ответа).
- Заголовок `Etag` по сути является механизмом валидации. Суть заключается в том, что сервер отправляет ресурс вместе с уникальным идентификатором (хеш-значением), при повторном запросе, если ресурс не изменился (хеш остался прежним), то используется кешированная версия.

#### 2. Аутентификация и Cookie

В заголовке HTTP можно настраивать специальные сессии и куки для быстрого входа по логину и паролю.

Заголовки, которые работают в связке по схеме "запрос-ответ" для защиты ресурсов:

- `WWW-Authenticate` - заголовок ответа от сервера, который сообщает клиенту, какая схема аутентификации требуется для доступа к ресурсу.
- `Authorization` - заголовок запроса от пользователя, который содержит тип аутентификации и учетные данные.

HTTP не сохраняет состояние между запросами, что делает невозможным отслеживание пользовательских сессий без дополнительных механизмов. Куки (Cookie) - механизм сохранения небольших порций данных на стороне клиента, которые автоматически отправляются браузером при последующих запросах к тому же домену. Для установления куки сервер использует заголовок `Set-Cookie`. При последующих запросах браузер автоматически включает заголовок `Cookie`.

Параметры управления жизненным циклом Cookie:

- `Expires` - абсолютная дата истечения срока.
- `Max-age` - время жизни с момента установки.

Типы Cookie по времени жизни:

- Сессионные куки. Не имеют параметров `Expires`/`Max-age`, удаляются при закрытии браузера, обычно используются для временных данных.
- Постоянные куки. Имеют указанный срок жизни, сохраняются между сессиями браузера, используются для настроек пользователя и аутентификации.

**Методы аутентификации в HTTP**:

1. Сессионная аутентификация на основе cookie. При аутентификации сервер создает сессию и сохраняет ID сессии в cookie, при последующих запросах сервер проверяет сессию по ID.
2. Простая схема с логином и паролем в закодированном виде (Basic). 
3. Аутентификация с использованием токенов (Bearer). Клиент получает токен после успешной аутентификации и отправляет его в заголовке с каждым запросом. В отличие от аутентификации на основе cookie сервер не хранит токены.
4. Схема с хешированием паролей (Digest). Является улучшением Basic схемы с защитой от перехвата пароля.

#### 3. Описание содержимого

Заголовки являются метаданными о теле сообщения:

- `Content-Type` - заголовок, который указывает тип передаваемых данных, что позволяет клиенту и серверу понять, как интерпретировать тело запроса/ответа.
- `Content-Length` - заголовок, который указывает размер тела сообщения в байтах.

#### 4. Межсайтовые запросы: CORS

CORS (Cross-Origin Resource Sharing) - это механизм, который использует HTTP-заголовки, чтобы дать веб-приложению, работающему на одном домене, право доступа к выбранным ресурсам с другого домена.

Заголовки со стороны сервера:

- `Access-Control-Allow-Origin` - заголовок, который указывает с каких доменов разрешен доступ.
- `Access-Control-Allow-Methods` - заголовок, который указывает разрешенные HTTP-методы, например, GET, POST, PUT, DELETE.
- `Access-Control-Allow-Headers` - заголовок, который указывает HTTP-заголовки, которые могут использоваться в запросе.

Со стороны клиента:

- `Access-Control-Request-Method` - заголовок, который сообщает серверу, какой HTTP-метод будет использоваться в основном запросе.

### REST 

REST (Representational State Transfer) - архитектурный стиль, предназначенный для создания распределенных систем. Основная идея - представление ресурсов через унифицированные интерфейсы, что упрощает взаимодействие между клиентами и серверами.

Ключевые принципы REST:

1. Единообразие интерфейса (Uniform Interface). Идентификация ресурсов через URI, манипуляция ресурсами через представления, самодостаточные сообщения.
2. Отсутствие состояния (Stateless). Каждый запрос содержит всю необходимую информацию, сервер не хранит состояние клиента между запросами, сессия хранится на клиенте.
3. Кешируемость. Ответы должны явно указывать возможность кэширования, использование HTTP-механизмов кэширования.
4. Клиент-серверная архитектура. Разделение ответственности: клиент отвечает за пользовательский интерфейс, а сервер отвечает за хранение данных и бизнес-логику.
5. Слоистая система (Layered System). Используются промежуточные серверы, такие как прокси, кеши и балансировщики. Скрывается сложность от клиента.
6. Код по требованию (Code on Demand). Поддержка загрузки и выполнения кода на стороне клиента.

Другие подходы к вызову функций:

- GraphQL. Язык запросов, позволяющий клиенту точно указать требуемые данные в одном запросе.
- RPC. Протокол удалённого вызова процедур, который вызывает метод, а не ресурс данных.
- SOAP. Протокол для обмена структурированной информацией в веб-сервисах с использованием XML.

### HTTPS

Протокол HTTPS (Hypertext Transfer Protocol Secure) представляет собой защищенную версию протокола HTTP, которая обеспечивает шифрование данных, передаваемых между клиентом и сервером. Безопасность обеспечивается криптографическими протоколами TLS или его предшественником SSL, которые создают зашифрованное соединение.

Ключевые особенности:

- Шифрование данных. Использование симметричного шифрования для эффективности.
- Аутентификация. Подтверждение подлинности сервера с помощью сертификатов.
- Целостность данных. Гарантия того, что данные не были изменены при передаче.

HTTPS сохраняет совместимость с протоколом HTTP, что означает, что веб-сайты и веб-приложения могут использовать обычные HTTP-запросы и методы, но в зашифрованном виде.

## gRPC

gRPC (Google Remote Procedure Calling) - система удалённого вызова процедур с открытым исходным кодом, разработанная компанией Google. Основное назначение - реализация взаимодействия между микросервисными архитектурами при разработке приложений. Построена на основе протокола HTTP/2 и использует Protocol Buffers (Protobuf) для сериализации данных. Преимуществом Protobuf является использование компактного бинарного формата.

Основные преимущества gRPC:

- Высокая производительность и низкая задержка. Бинарный Protobuf и эффективность HTTP/2 делают gRPC одним из самых быстрых способов связи между сервисами.
- Строгая типизация. Использование Protobuf обеспечивает строгую типизацию данных и контрактов между клиентом и сервером.
- Встроенная генерация кода. Мощная кодогенерация для множества языков значительно ускоряет разработку.
- Поддержка потоковой передачи. Двунаправленный поток данных между сервером и клиентом, что подходит для приложений реального времени.
- Надежность и экосистема. gRPC отлично подходит для построения сложных распределенных систем, благодаря поддержке дедлайнов/таймаутов, метаданных и аутентификации.

### Сравнение gRPC и REST/HTTP API

| Аспект              | gRPC                                  | REST/HTTP API                                  |
|---------------------|---------------------------------------|------------------------------------------------|
| Формат сообщений    | Бинарный (Protobuf)                   | Структурированный (JSON/XML)                   |
| Контракт API        | Строго типизированный                 | Неформальный (OpenAPI)                         |
| Скорость обмена     | Высокая (двунаправленный поток)       | Средняя, обмен однонаправленный (запрос-ответ) |
| Поддержка браузеров | Ограниченная                          | Полная                                         |
| Производительность  | Высокая (данные сериализованы)        | Средняя (данные требуют сериализации)          |
| Формат сообщений    | Бинарный (Protobuf)                   | Структурированный (JSON/XML)                   |

gRPC решает проблемы производительности, эффективности сетевого взаимодействия, строгости контрактов, которые являются узкими местами в высоконагруженных микросервисных архитектурах при использовании классического REST подхода.

### Поддерживаемые типы взаимодействий RPC

В системе удалённого вызова процедур gRPC поддерживаются несколько типов взаимодействий (RPC):

1. Унарный (Unary). Базовая и простая модель. Клиент отправляет один запрос серверу и получает в ответ одно сообщение. Аналогично обычному HTTP.
2. Серверный поток (Server streaming). Клиент отправляет один запрос серверу, после чего сервер начинает отправлять поток ответов. Используется в сценариях, в которых серверу требуется отправить большое количество данных или передать постоянно обновляемую информацию.
3. Клиентский поток (Client streaming). Клиент отправляет поток данных серверу, после завершения отправки потока клиент ожидает ответ от сервера. Например, загрузка файла.
4. Двусторонний поток (Bidirectional streaming). Клиент и сервер обмениваются потоками данных в обоих направлениях. Подходит для чатов и онлайн-игр.

### Protocol Buffers

Protobuf - это механизм сериализации структурированных данных, который является языконезависимым, платформонезависимым и расширяемым. Он значительно более эффективен по сравнению с JSON/XML как по размеру сообщений, так и по скорости сериализации/десериализации.

#### Пример описания .proto-файла и генерация кода

`.proto`-файлы используются для определения структуры данных и сервисов. Базовыми элементами являются сообщения и сервисы.

Пример `.proto` (`user_service.proto`):
```
syntax = "proto3";

package user_service;

// Cообщения
message GetUserRequest {
  int32 user_id = 1;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message UserResponse {
  int32 id = 1;
  string name = 2;
  string email = 3;
  string status = 4;
}

// Сервис для работы с пользователями
service UserService {
  rpc GetUser(GetUserRequest) returns (UserResponse);
  
  rpc CreateUser(CreateUserRequest) returns (UserResponse);
  
  rpc ListUsers(GetUserRequest) returns (stream UserResponse);
}
```

После описания `.proto` можно сгенерировать код, например на Python:
```
python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. user_service.proto
```

После выполнения данного скрипта появляются два файла:

- `user_service_pb2.py` - классы сообщений.
- `user_service_pb2_grpc.py` - классы для gRPC сервиса.

#### Различие версий proto

В нашем примере мы использовали proto3, которая является последней версией. Proto3 - это упрощенная и более производительная версия proto2. 

Сравнение версий proto:

| Критерий                  | proto2                             | proto3                     |
|---------------------------|------------------------------------|----------------------------|
| **Обязательные поля**     | Есть (`required`)                  | Только поля `optional`     |
| **Значения по умолчанию** | Можно задать                       | Используются фиксированные |
| **Неизвестные поля**      | Сохраняются                        | Отбрасываются              |
| **Поддержка JSON**        | Ограниченная                       | Нативная                   |
| **Расширения**            | ПОддерживаются                     | Заменены на тип `Any`      |
| **Производительность**    | Хорошая (лучше текстовых форматов) | Лучшая (оптимизирована)    |

#### Типы данных в Protobuf и их сопоставление с другими языками

| Тип Protobuf | Java | Python | Go | C++ |
|-------------|------|--------|-----|-----|
| double | double | float | float64 | double |
| float | float | float | float32 | float |
| int32 | int | int | int32 | int32 |
| int64 | long | int | int64 | int64 |
| uint32 | int | int | uint32 | uint32 |
| uint64 | long | int | uint64 | uint64 |
| bool | boolean | bool | bool | bool |
| string | String | str | string | std::string |
| bytes | ByteString | bytes | []byte | std::string |
| enum | enum | enum | int | enum |
| message | Класс | Класс | Структура | Класс |

Помимо скалярных типов, Protobuf поддерживает структурированные типы:

- **Вложенные сообщения**. Определение сообщения внутри другого.
- **Перечисления (enum)**. Определение списка именованных констант.
- **Повторяющиеся поля (repeated)**. Эквивалент массивов или списков.
- **Maps**. Ассоциативные массивы (словари), представленные как `map<key_type, value_type>`.
- **OneOf**. Поле, которое может содержать одно из нескольких различных типов сообщений.

#### Сериализации и десериализации

Процесс сериализации и десериализации данных в gRPC автоматически обрабатывается сгенерированным кодом:

1. **Сериализация**: одна из сторон создает объект сообщения, заполняет его данными, и сгенерированный код преобразует объект в бинарный формат Protobuf.
2. **Передача**: сериализованные данные передаются по HTTP/2 соединению.
3. **Десериализация**: на второй стороне бинарные данные преобразуются обратно в объект.

Каждая сторона выполняет как сериализацию (исходящих сообщений), так и десериализацию (входящих сообщений).

### Обработка ошибок в gRPC

gRPC использует стандартизированную систему статусов для обработки ошибок. Список кодов статусов состояний:

| Статус | Код | Описание |
|--------|-----|----------|
| OK | 0 | Успешное выполнение |
| CANCELLED | 1 | Операция отменена |
| UNKNOWN | 2 | Неизвестная ошибка |
| INVALID_ARGUMENT | 3 | Неверный аргумент |
| DEADLINE_EXCEEDED | 4 | Превышен лимит времени |
| NOT_FOUND | 5 | Ресурс не найден |
| ALREADY_EXISTS | 6 | Ресурс уже существует |
| PERMISSION_DENIED | 7 | Отказано в доступе |
| RESOURCE_EXHAUSTED | 8 | Ресурс исчерпан |
| FAILED_PRECONDITION | 9 | Не выполнено предусловие |
| ABORTED | 10 | Операция прервана |
| OUT_OF_RANGE | 11 | Выход за пределы диапазона |
| UNIMPLEMENTED | 12 | Метод не реализован |
| INTERNAL | 13 | Внутренняя ошибка |
| UNAVAILABLE | 14 | Сервис недоступен |
| DATA_LOSS | 15 | Потеря данных |
| UNAUTHENTICATED | 16 | Не аутентифицирован |

Более подробно можете изучить статусы состояний [в официальной документации](https://grpc.io/docs/guides/status-codes/).

Для передачи дополнительной информации об ошибках используется [Rich Error Model](https://google.aip.dev/193#error_model). Вместо одного кода ошибки и сообщения, система позволяет передавать несколько типов метаданных в машиночитаемом формате.

### Метаданные

Метаданные в gRPC - это пары ключ-значение, которые передаются вместе с RPC-вызовом, но не являются частью основного сообщения (запроса/ответа). Они аналогичны HTTP-заголовкам.

Метаданные используются для:

- Аутентификации и авторизации.
- Управляющей информации (дедлайны, кодировки).
- Отладочной информации (трассировка).

Пример отправки метаданных на Go:
```go
ctx := context.Background()
md := metadata.Pairs(
    "authorization", "Bearer token123",
    "request-id", "123456",
)
ctx = metadata.NewOutgoingContext(ctx, md)

// Вызов RPC с метаданными
response, err := client.GetUser(ctx, &pb.GetUserRequest{UserId: 1})
```

### Аутентификация 

gRPC поддерживает несколько механизмов аутентификации:

#### 1. SSL/TLS

gRPC имеет интеграцию с SSL/TLS и поддерживает использование SSL/TLS для аутентификации сервера и шифрования всех данных, которыми обмениваются клиент и сервер.

Пример аутентификации сервера, т. е. клиент хочет аутентифицировать сервер и зашифровать все данные, на C++:
```C++
// Создание обычного объекта SSL ChannelCredentials.
auto channel_creds = grpc::SslCredentials(grpc::SslCredentialsOptions());
// Создание канала с использованием учетных данных, созданных на предыдущем шаге.
auto channel = grpc::CreateChannel(server_name, channel_creds);
// Создание объекта-заглушки (stub) на канале.
std::unique_ptr<Greeter::Stub> stub(Greeter::NewStub(channel));
// Выполнение реальных RPC-вызовов через заглушку.
grpc::Status s = stub->sayHello(&context, *request, response);
```

#### 2. Token-based

Аутентификация на основе токенов включает в себя 3 этапа:

- Генерация токена на стороне клиента.
- Создание учетных данных с помощью токена.
- Проверка токена на стороне сервера.

Пример аутентификации на основе токенов Google:
```C++
auto creds = grpc::GoogleDefaultCredentials();
// Создание канала, заглушки и выполнение вызовов RPC
auto channel = grpc::CreateChannel(server_name, creds);
std::unique_ptr<Greeter::Stub> stub(Greeter::NewStub(channel));
grpc::Status s = stub->sayHello(&context, *request, response);
```

#### 3. mTLS

Mutual TLS (mTLS) - механизм аутентификации, который добавляет дополнительный уровень безопасности, проверяя идентичность как клиента, так и сервера.

Процесс аутентификации:

- Клиент отправляет запрос на доступ к защищенной информации с сервера.
- Сервер отправляет клиенту свой сертификат.
- Клиент проверяет полученный сертификат через центр сертификации.
- Если проверка прошла успешно, то клиент отправляет свой сертификат на сервер.
- Сервер аналогичным образом проверяет сертификат клиента.
- После успешного завершения проверки сервер выдает разрешение на доступ к защищенным данным.

### Авторизация

gRPC предоставляет гибкие точки расширения, куда можно встроить логику авторизации. Ключевые подходы можно разделить на несколько категорий:

1. На уровне вызова с использованием метаданных. Клиент передает токен доступа в метаданных каждого вызова. Сервер затем перехватывает этот токен, проверяет его и на основе содержащейся в нем информации принимает решение об авторизации. Сервер использует interceptor (перехватчик) для перехвата всех входящих вызовов.
2. Авторизация на уровне сервера. Можно проверять права непосредственно в реализации каждого метода сервиса. Подход проще, но ведёт к дублированию кода.
3. Интеграция со сторонними системами. В этой схеме gRPC-сервер не содержит код авторизации, а доверяет сторонним системам.

### Инструменты для отладки gRPC 

#### 1. GRPCurl

GRPCurl - это инструмент командной строки для тестирования и взаимодействия с сервисами gRPC.

Некоторые примеры использования:

- Просмотр списка сервисов:

```
$ grpcurl localhost:<port> list
```

- Описание сервисов:

```
$ grpcurl localhost:<port> describe
```

- Пример вызова сервиса:

```
$ grpcurl -d '{ \"name\": \"World\" }' localhost:<port> greet.Greeter/SayHello
```

#### 2. gRPCUI

gRPCUI - это интерактивный пользовательский веб-интерфейс для gRPC. Он предлагает графический интерфейс для обнаружения и тестирования служб gRPC.

Для использования нужно выполнить команду:
```
grpcui localhost:<port>
```

После выполнения команды, откроется окно в браузере с интерактивным пользовательским интерфейсом.

### Контекст (Context) в gRPC

Контекст в gRPC - это объект, который передается в каждый RPC-вызов и содержит метаданные, информацию о дедлайнах, механизмы отмены и другие данные, связанные с выполнением запроса.

Функции контекста:

- Передача метаданных между клиентом и сервером.
- Управление временем выполнения.
- Хранение пользовательских данных.
- Механизм отмены операций.

Пример использования контекста (клиентская сторона):
```go
// Создание контекста с таймаутом
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

// Добавление метаданных
ctx = metadata.NewOutgoingContext(ctx, metadata.Pairs(
    "authorization", "Bearer token123",
))

response, err := client.GetUser(ctx, request)
```

Серверная сторона:
```go
func (s *server) GetUser(ctx context.Context, req *pb.Request) (*pb.Response, error) {
    // Проверка дедлайна
    if ctx.Err() == context.DeadlineExceeded {
        return nil, status.Error(codes.DeadlineExceeded, "Timeout")
    }
    
    // Чтение метаданных
    md, _ := metadata.FromIncomingContext(ctx)
    token := md.Get("authorization")
    
    return &pb.Response{...}, nil
}

```
