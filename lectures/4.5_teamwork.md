# Лекция 20 - Работа в команде

Командная разработка - это слаженная работа над общим проектом, эффективность которой значительно повышается с использованием распределенных систем контроля версий (Git) и удаленных репозиториев (например, GitHub).

## Правила создания коммитов

#### 1. Не следует отправлять ненужные пробелы.

Git предоставляет простой способ проверки наличия ненужных пробелов - команда `git diff --check`, которая выведет список ненужных пробелов:
```
$ git diff --check
app.py:4: trailing whitespace.
+@app.route('/')  ^M
app.py:8: trailing whitespace.
+if __name__ == '__main__':  ^M
```
В примере выше в двух строках имеются ненужные пробелы в конце строк.

#### 2. Один коммит - одна задача.

Следует разделять проделанную работу по принципу минимум один коммит на задачу, давая полезные комментарии к каждому из них. Состояние проекта в конце ветки не зависит от количества сделанных коммитов, так как все изменения добавляются в один момент, поэтому старайтесь облегчить задачу вашим коллегам, которые будут просматривать ваши изменения.

#### 3. Качественные сообщения к коммитов.

Как правило, сообщения должны начинаться кратким однострочным описанием (не более 50 символов), затем должна идти пустая строка, после которой следует более детальное описание. Приветствуется в детальном описании наличие причины внесения изменений и сравнение с текущей реализацией. Сообщение коммита следует писать в императива (например, "Fix bug", "Add feature"). 

## Pull Request (PR) / Merge Request (MR)

PR/MR - это механизм в системах контроля версий (GitHub, GitLab), который позволяет разработчикам предложить изменения в коде и обсудить их перед слиянием. Pull Request и Merge Request - это одно и то же понятие, но используемое в разных системах контроля версий.

При работе в команде перед тем, как вносить изменения в проект, обычно требуется одобрение от коллег, поэтому pull request выступает в роли инструмента для код-ревью. Разработчик создаёт ветку, вносит свои изменения, затем создает запрос на слияние этой ветки с основной веткой проекта. Таким образом, PR является ключевой практикой для поддержания качества кода и документирования изменений в проекте.

Пример:
```
$ git checkout -b feature
Switched to a new branch 'feature'

$ git branch
* feature
  main

$ git add .

$ git commit

$ git push git_test feature
...
To https://github.com/<git_username>/git_test
 * [new branch]      feature -> feature
```

Теперь, когда мы зайдём на страницу репозитория `https://github.com/<git_username>/git_test`, мы увидим, что GitHub заметил изменения и предлагает открыть запрос на слияние (PR) с помощью большой зелёной кнопки. После нажатия на кнопку, появится экран ввода заголовка и описания PR, также мы увидим список коммитов и предпросмотр изменений, вносимых этими коммитами.

## Git Flow

Git Flow - модель ветвления, которая определяет строгие правила создания веток и их слияния. Методология Git Flow необходима, чтобы структурировать работу с кодом на всех стадиях разработки.

### Структура веток

Основные ветки в Git Flow:
1. main/master - ветка, в которой хранится самая последняя и готовая к релизу версия кода.
2. develop - ветка, которая объединяет изменения в проекте (в этой ветке ведется активная разработка).
3. feature - ветка для разработки новых функций.
4. release - ветка, в которой исправляют последние ошибки и подготавливают релиз.
5. hotfix - ветка, которая используется для быстрого исправления ошибок в ветке main.

### Достоинства и недостатки

Плюсы:
- Удобный процесс для код-ревью.
- Подходит для релизов 1 раз в 1-2 недели.
- Хорошо подходит для работы с несколькими релизами и командами.

Минусы:
- Сложно делать много частых релизов.
- Может возникать больше merge-конфликтов.

### Альтернативные подходы

#### 1. GitHub Flow (упрощенный подход)

Используется одна ветка - master. От неё создаются ветки для отдельных задач, которые после выполнения обратно сливаются в master через PR. Данная модель значительно удобнее при частых релизах.

#### 2. Trunk-Based Development

Основная разработка ведется в одной общей ветке - trunk (тот же main/master). Все изменения вносятся напрямую в эту центральную ветку или через небольшие короткоживущие ветки, которые быстро объединяются обратно. Основная ветка всегда находится в рабочем состоянии и готова к релизу.

## Code Review

Код-ревью - систематическая проверка кода одним или несколькими разработчиками для выявления ошибок, улучшения качества кода, обмена знаниями и соблюдения стандартов.

### Как проводиться Code Review через Git

Первым делом принято внимательно ознакомиться с описанием pull request, на данном этапе необходимо понять, что именно должно быть сделано, какова цель изменений. Далее анализируются изменения, которые вносит PR, на этом этапе проверяется корректность логики, читаемость кода, соответствие стандартам и требованиям. При анализе ревьювер оставляет комментарии, задает вопросы и предлагает улучшения. После чего, автор вносит правки на основе комментариев. После внесения правок и утверждения изменений, производится слияние (merge).

### Практики Code Review

Подготовка к код-ревью:
1. Определение целей. Какую проблему должны решать вносимые изменения?
2. Проверка компиляции и тестирования.
3. Ознакомление со связанной задачей, в которой описывается проблема и ресурсы для решения.

Проведение код-ревью:
1. Сравнение со стандартами. Стоит полагаться на стандарты и требования, принятые в команде.
2. Конструктивная обратная связь. Важно сосредоточиться на улучшении кода, а не на осуждении автора.
3. Своевременность. Стоит давать обратную связь в оговоренные командой сроки.
4. Тестирование. В сложных задачах может быть полезно сказать ветку и запустить код/тесты. Это помогает найти проблемы, не видимые глазу в критически важных ситуациях.
