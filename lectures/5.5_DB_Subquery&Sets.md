# Лекция 26 - Подзапросы и работа с множествами

## Подзапросы

Подзапрос - это запрос, содержащийся в другом SQL-выражении (основном). Подзапрос заключен в круглые скобки и выполняется до основного выражения. По завершении подзапрос возвращает временную таблицу, которая выгружается из памяти после завершения основного запроса.

Рассмотрим пример подзапроса:
```
SELECT student_id, first_name, last_name, average_grade
FROM students
WHERE group_id = (
	SELECT group_id
	FROM students
	GROUP BY group_id
	ORDER BY AVG(average_grade) DESC
	LIMIT 1
)
ORDER BY average_grade DESC;

 student_id | first_name | last_name | average_grade
------------+------------+-----------+---------------
          8 | Ольга      | Зайцева   |          5.00
          2 | Мария      | Сидорова  |          4.90
          6 | Анна       | Морозова  |          4.80
          5 | Дмитрий    | Соколов   |          4.50
          1 | Иван       | Петров    |          4.30
          9 | Павел      | Козлов    |          4.20
          7 | Сергей     | Волков    |          3.90
(7 строк)
```

Данный запрос формирует результирующий набор из студентов группы, которая имеет наибольший средний балл. Id группы с наивысшим средним баллом определяется подзапросом.

В данном примере подзапрос возвращает одну строку и один столбец, что удобно, когда результат подзапроса используется как условие фильтрации для внешнего запроса. Такой тип подзапросов называется скалярным подзапросом (scalar subquery). Также подзапросы, как и обычные запросы, могут возвращать несколько строк и столбцов.

Также запросы разделяют по признаку автономности. Запросы, которые могут выполняться независимо и полностью самодостаточны, называются **несвязанные** или **некоррелированные** (noncorrelated subqueries), а запросы, которые ссылаются на столбцы основного выражения и не могут выполняться независимо, называют **связанными** или **коррелированными** (correlated subqueries).

### Некоррелированные подзапросы

Пример выше является некоррелированным подзапросом. Он может выполняться самостоятельно и не использует ничего из основного запроса.

Если бы подзапрос возвращал несколько строк, то его нельзя было бы использовать с оператором сравнения. Однако мы могли бы использовать оператор `IN`, который был рассмотрен во второй лекции по БД. Допустим мы захотим выделить не одну группу, а две:
```
SELECT group_id, student_id, first_name, last_name, average_grade
FROM students
WHERE group_id IN (
	SELECT group_id
	FROM students
	GROUP BY group_id
	ORDER BY AVG(average_grade) DESC
	LIMIT 2
)
ORDER BY average_grade DESC;
```

Также можно использовать оператор `ALL`, который позволяет сравнивать одиночное значение с каждым элементом. Используя `ALL` и оператор неравенства `<>`, мы можем вывести студентов, не принадлижащим к двум группам, возвращаемым подзапросом:

```
SELECT group_id, student_id, first_name, last_name, average_grade
FROM students
WHERE group_id <> ALL (
	SELECT group_id
	FROM students
	GROUP BY group_id
	ORDER BY AVG(average_grade) DESC
	LIMIT 2
)
ORDER BY average_grade DESC;

```

Оператор `ANY` также позволяет сравнивать одиночное значение с набором, однако он возвращает `TRUE`, если было хотя бы одно совпадение (`TRUE`):

```
SELECT group_id, student_id, first_name, last_name, average_grade
FROM students
WHERE group_id = ANY (

	WHERE average_grade > 4.8
)
ORDER BY average_grade DESC;
```

### Коррелированные подзапросы

Зависимость коррелированных запросов проявляется в том, что они ссылаются на один или несколько столбцов основного запроса. Допустим, нам необходимо выделить информацию о студентах и количество курсов, которые они проходят:
```
SELECT s.student_id, s.first_name, s.last_name, (
	SELECT COUNT(*) AS cnt_courses
	FROM student_courses AS s_c
	WHERE s.student_id = s_c.student_id
	GROUP BY s_c.student_id
) AS count_courses
FROM students AS s;

 student_id | first_name | last_name | count_courses
------------+------------+-----------+---------------
          1 | Иван       | Петров    |             2
          2 | Мария      | Сидорова  |             3
          5 | Дмитрий    | Соколов   |             2
          6 | Анна       | Морозова  |             3
          7 | Сергей     | Волков    |             2
          8 | Ольга      | Зайцева   |             3
          9 | Павел      | Козлов    |             2
          3 | Алексей    | Кузнецов  |             3
         10 | Наталья    | Новикова  |             2
         11 | Андрей     | Медведев  |             2
         12 | Екатерина  | Попова    |             3
         13 | Артем      | Михайлов  |             2
         14 | Виктория   | Федорова  |             3
          4 | Елена      | Васильева |             2
         15 | Максим     | Алексеев  |             1
         16 | Алина      | Лебедева  |             3
         17 | Игорь      | Степанов  |             2
         18 | Татьяна    | Ковалева  |             3
         19 | Роман      | Семенов   |             2
(19 строк)
```

В данном примере запрос ссылается на `student_id` из основного запроса. В отличие от некоррелированных подзапросов, которые выполняются однократно, коррелированный подзапрос выполняется для каждой строки внешнего запроса.

И в этом заключается его главный недостаток - потенциальные проблемы с производительностью. Чем больше строк в таблице `students`, тем больше раз выполняется подзапрос. Заметьте, что мы уже решали подобную проблему с помощью `JOIN` (в прошлой лекции мы рассчитывали количество студентов на каждом курсе), при таком подходе у нас получается более эффективный и быстрый запрос.

Кореллированные подзапросы часто используются в выражениях `UPDATE`, `DELETE` и `INSTERT`. Допустим, мы хотим периодически очищать информацию о курсах, на которых нет студентов. В таком случае будет удобно использовать коррелированный подзапрос:
```
DELETE FROM courses as c
WHERE NOT EXISTS (
	SELECT 1
	FROM student_courses as s_c
	WHERE c.course_id = s_c.course_id
)
```

`EXISTS` возвращает `TRUE`, если в подзапросе есть хотя бы одна строка, `NOT EXISTS` соответственно возвращает `TRUE`, если строк нет. Используется `SELECT 1`, поскольку для `EXISTS` важно лишь существование строк, а не их содержимое.

### Обобщённое табличное выражение (CTE)

Обобщенное табличное выражение (CTE, Common Table Expressions) - это временный результирующий набор, который создаётся в рамках выполнения запроса. CTE можно представить как именованный подзапрос, который также существует в рамках одного запроса.

Для создания CTE используется ключевое слово `WITH`, а его тело располагается перед запросом. Также как и некоррелированный подзапрос CTE выполняется перед основным запросом, поэтому оно доступно для основного запроса.

Рассмотрим пример использования CTE:
```
WITH hard_course AS (
    SELECT 
        c.course_id
    FROM courses AS c
    JOIN student_courses AS s_c
		ON c.course_id = s_c.course_id
    GROUP BY c.course_id
    ORDER BY AVG(s_c.grade) ASC
    LIMIT 2
)

SELECT s.student_id,
	s.first_name,
	s.last_name,
	s.group_id,
	ROUND(AVG(s_c.grade), 1) AS avg_grade
FROM students AS s
JOIN student_courses AS s_c
	ON s.student_id = s_c.student_id
JOIN hard_course AS h_c
	ON s_c.course_id = h_c.course_id
GROUP BY s.student_id;

 student_id | first_name | last_name | group_id | avg_grade
------------+------------+-----------+----------+-----------
          4 | Елена      | Васильева |        3 |       4.5
         15 | Максим     | Алексеев  |        3 |       4.0
         16 | Алина      | Лебедева  |        3 |       4.5
         17 | Игорь      | Степанов  |        3 |       4.0
         18 | Татьяна    | Ковалева  |        3 |       5.0
         19 | Роман      | Семенов   |        3 |       3.0
(6 строк)
```

В CTE `hard_course` мы находим 2 курса, на которых студенты имеют наименьшие оценки. В основном запросе мы выводим средние оценки студентов по самым сложным курсам.

## Работа с множествами

Операции с множествами осуществляются путем помещения оператора работы со множествами (set operator) между выражениями `SELECT`:
```
SELECT VARCHAR 'Андрей', VARCHAR '18'
UNION
SELECT VARCHAR 'Иван', VARCHAR '19';

 varchar | varchar
---------+---------
 Андрей  | 18
 Иван    | 19
(2 строки)
```

Каждый запрос (`SELECT`) формирует таблицу с двумя столбцами и одной строкой, оператор работы с множествами (`UNION`) объединяет все строки таблиц. В итоге конечная таблица содержит две строки. Такие запросы, которые объединяют несколько независимых запросов, называют составными.

### Операторы работы с множествами

#### UNION

Операторы `UNION` и `UNION ALL` комбинируют несколько таблиц. Оператор `UNION ALL` не проверяет дублирующиеся значения, в результирующем наборе будет столько строк, сколько во всех исходных таблицах в сумме:
```
SELECT student_id, first_name, last_name
FROM students
UNION ALL
SELECT student_id, first_name, last_name
FROM students;

 student_id | first_name | last_name
------------+------------+-----------
          1 | Иван       | Петров
          2 | Мария      | Сидорова
          5 | Дмитрий    | Соколов
          6 | Анна       | Морозова
          7 | Сергей     | Волков
          8 | Ольга      | Зайцева
...
(38 строк)
```

Оператор `UNION` не включает дублирующиеся значения:
```
SELECT student_id, first_name, last_name
FROM students
UNION
SELECT student_id, first_name, last_name
FROM students;

 student_id | first_name | last_name
------------+------------+-----------
          7 | Сергей     | Волков
          4 | Елена      | Васильева
         18 | Татьяна    | Ковалева
          5 | Дмитрий    | Соколов
         19 | Роман      | Семенов
         16 | Алина      | Лебедева
...
(19 строк)
```

#### INTERSECT

Оператор `INTERSECT` выполняет операцию пересечения, возвращая общие строки для двух запросов. Дубликаты при этом удаляются.

Пример:
```
SELECT student_id, first_name, last_name, average_grade
FROM students
WHERE average_grade >= 4.5

INTERSECT

SELECT student_id, first_name, last_name, average_grade
FROM students AS s
INNER JOIN groups AS g
	ON g.group_id = s.group_id
WHERE g.enrollment_year = (
	SELECT MIN(enrollment_year)
	FROM groups
);

 student_id | first_name | last_name | average_grade
------------+------------+-----------+---------------
         18 | Татьяна    | Ковалева  |          4.70
          4 | Елена      | Васильева |          4.60
(2 строки)
```

Первый запрос возвращает множество отличников (средний балл выше 4.5), второй запрос возвращает множество студентов самого старшего курса. Пересечение двух запросов даёт студентов, которые одновременно являются и отличниками, и принадлежат самому старшему курсу.

Оператор `INTERSECT ALL` работает похоже, но сохраняет дубликаты. Если в первом запросе строка встречается 3 раза, а во втором - 2 раза, то в результате она появится 2 раза (минимальное количество вхождений):
```
WITH st_duplicates42 AS (
	SELECT *
    FROM students
    WHERE student_id = 4
	
	UNION ALL

	SELECT *
    FROM students
), st_duplicates43 AS (
	SELECT *
    FROM students
    WHERE student_id = 4

	UNION ALL
	
	SELECT *
    FROM students
    WHERE student_id = 4
	
	UNION ALL

	SELECT *
    FROM students
)

SELECT student_id, first_name, last_name, average_grade
FROM st_duplicates42
WHERE average_grade >= 4.5

INTERSECT ALL

SELECT student_id, first_name, last_name, average_grade
FROM st_duplicates43 AS s
INNER JOIN groups AS g
	ON g.group_id = s.group_id
WHERE g.enrollment_year = (
	SELECT MIN(enrollment_year)
	FROM groups
);

 student_id | first_name | last_name | average_grade
------------+------------+-----------+---------------
         18 | Татьяна    | Ковалева  |          4.70
          4 | Елена      | Васильева |          4.60
          4 | Елена      | Васильева |          4.60
(3 строки)
```

К предыдущему запросу мы добавили 2 CTE, в которых содержатся дубликаты студента с id = 4 (студент встречается в одном 2 раза, в другом 3 раза), в результирующий набор студент с id = 4 попал всего 2 раза, что соответствует правилу `INTERSECT ALL`.

#### EXCEPT

Оператор `EXCEPT` выполняет операцию исключения, возвращая строки первой таблицы с вычетом перекрытий со второй таблицы. Также есть оператор `EXCEPT ALL`, который удаляет из первого множества один экземпляр дубликата для каждого экземпляра дубликата второго множества. 

Для демонстрации рассмотрим пример. Сформируем снова два CTE:
```
WITH A AS (
    SELECT 1 AS int_
    UNION ALL
    SELECT 1
    UNION ALL
    SELECT 1
    UNION ALL
    SELECT 2
    UNION ALL
    SELECT 5
), B AS (
	SELECT 1 AS int_
    UNION ALL
    SELECT 1
)
```

Множество А содержит: 1, 1, 1, 2, 5. Множество B содержит: 1, 1.

`EXCEPT`:
```
SELECT *
FROM A

EXCEPT

SELECT *
FROM B;

 int_
------
    5
    2
(2 строки)
```

В результате операции `EXCEPT` мы убираем все единицы из множества А, хотя их там 3, а в множестве B всего 2.

`EXCEPT ALL`:
```
SELECT *
FROM A

EXCEPT ALL

SELECT *
FROM B;

 int_
------
    1
    5
    2
(3 строки)
```

В результате `EXCEPT ALL` одна единица в результирующем множестве всё же остаётся, что подтвержает описанное выше правило.

### Сортировка составного запроса

Если требуется сортировать результаты составного запроса, после последнего входящего в него запроса можно добавить блок `ORDER BY`:
```
SELECT *
FROM A

EXCEPT ALL

SELECT *
FROM B
ORDER BY int_;

 int_
------
    1
    2
    5
(3 строки)
```

### Приоритет операторов работы со множествами

| Приоритет | Операторы |
|-----------|-----------|
| Высокий | `INTERSECT`, `INTERSECT ALL` |
| Низкий | `UNION`, `UNION ALL`, `EXCEPT`, `EXCEPT ALL` |

Порядок выполнения:
- `INTERSECT` выполняется первым.
- `UNION` и `EXCEPT` имеют одинаковый приоритет и выполняются сверху вниз. `ALL`-версии подчиняются тем же правилам.
- Скобки позволяют явно задать порядок выполнения, изменяя стандартный приоритет операций.
