# Лекция 9 - Протоколы транспортного уровня - TCP, UDP

## Транспортный уровень

Транспортный уровень располагается между сетевым и прикладным уровнем стека протоколов TCP/IP. В то время как задачей сетевого уровня является передача данных между произвольными узлами сети, задача транспортного уровня заключается в передаче данных между любыми прикладными процессами, выполняющимися на любых узлах сети.

Основными протоколами транспортного уровня являются протокол управления передачей TCP (Transmission Control Protocol) и протокол пользовательских дейтаграмм UDP (User Datagram Protocol).

## UDP

UDP - дейтаграммный протокол транспортного уровня, который обеспечивает минимальный механизм передачи данных между приложениями. Задачей протокола транспортного уровня UDP является передача данных между прикладными процессами без гарантии их доставки. Он не устанавливает логического соединения, не нумерует и не упорядочивает пакеты.

Протокол не гарантирует, что в процессе передачи его дейтаграммы не могут быть потеряны, продублированы или прийти не в том порядке, в котором они были отправлены. Термин без гарантии доставки говорит о том, что средствами самого протокола невозможно убедиться, что данные корректно получены адресатом.

С другой стороны, функциональная простота протокола UDP обуславливает простоту его алгоритма, компактность и высокое быстродействие. Поэтому те приложения, где реализован собственный, достаточно надежный, механизм обмена сообщениями, предпочитают для непосредственной передачи данных по сети использовать менее надежные, но более быстрые средства транспортировки.

UDP применяется в приложениях, где важнее скорость, чем надежность:
- Потоковые мультимедиа.
- Игровые приложения.
- Системы реального времени.
- DNS-запросы.

### Заголовок UDP

Заголовок UDP имеет фиксированный размер - 8 байт и состоит из 4 полей:

|Биты|0 - 15|16 - 31|
|----|------|-------|
|0-31| Порт отправителя (Source port) | Порт получателя (Destination port) |
|32-63|	Длина дейтаграммы (Length)| Контрольная сумма (Checksum)|

Поля заголовка:
- **Порт отправителя**. В этом поле указывается номер порта отправителя. Предполагается, что это значение задаёт порт, на который при необходимости будет посылаться ответ. В противном же случае значение должно быть равным 0.
- **Порт получателя**. Это поле обязательно и содержит порт получателя.
- **Длина дейтаграммы**. Поле, задающее длину всей датаграммы (заголовка и данных) в байтах. Минимальная длина равна длине заголовка — 8 байт. Максимальная длина за вычетом минимального IP-заголовка будет 65535 - 20 = 65515 байт.
- **Контрольная сумма**. Поле контрольной суммы используется для проверки заголовка и данных на ошибки. Если сумма не сгенерирована передатчиком, то поле заполняется нулями. Поле не является обязательным для IPv4.

### Ненадежность UDP

Из-за недостатка надёжности приложения UDP должны быть готовы к некоторым потерям, ошибкам и дублированиям. Некоторые из них (например, TFTP) могут при необходимости добавить элементарные механизмы обеспечения надёжности на прикладном уровне. Но чаще такие механизмы не используются UDP-приложениями и даже мешают им. Если приложению необходим высокий уровень надёжности, то можно использовать TCP.

## TCP

Для обеспечения надёжной доставки данных на транспортном уровне используется протокол TCP (Transmission Control Protocol - протокол управления передачей) - надежный потоковый протокол, требующий создания логических соединений. Надёжность TCP обеспечивает механизм подтверждения приёма с повторной передачей (Positive Acknowledgment with Retransmission, PAR). Система с PAR повторяет отправку данных до тех пор, пока не получит от адресата подтверждение, что данные получены. 

TCP используется в приложениях, где важна надежность передачи данных:
- Веб-браузеры (HTTP/HTTPS).
- Электронная почта (POP3, SMTP, IMAP).
- Файловые передачи (FTP, SFTP).
- Удаленный доступ (SSH, Telnet).

### Сегменты TCP и его заголовок

Сегмент TCP — это единица передачи данных в протоколе TCP, состоящая из заголовка и данных.

Заголовок TCP имеет переменный размер от 20 до 60 байт и содержит следующие поля:

- Порт источника. Идентифицирует приложение клиента, с которого отправлены пакеты.
- Порт назначения. Идентифицирует порт, на который отправлен пакет.
- Порядковый номер (SN). Измеряется в байтах, идентифицирует номер первого байта данных в сегменте.
- Номер подтверждения (ACK SN). Если установлен флаг ACK, то это поле содержит порядковый номер байта, который отправитель данного сегмента желает получить. Это означает, что все предыдущие байты были успешно получены.
- Длина заголовка (Data offset). Длина заголовка определяет смещение полезных данных относительно начала сегмента.
- Флаги. Управляющие биты, например, ACK - подтверждение, SYN - синхронизация.
- Размер окна. Количество байт данных, после передачи которых отправитель ожидает подтверждения от получателя, что данные получены.
- Контрольная сумма. Служит для проверки целостности заголовка и данных.
- Указатель важности. Указывает на срочные данные.
- Опции. Имеет переменную длину и используется для вспомогательных задач. Например, при выборе максимального размера сегмента.
- Фиктивное поле (Padding). Используется для доведения размера заголовка до целого числа 32-битовых слов.

### TCP Handshake

TCP handshake (рукопожатие) - это процесс установления соединения между клиентом и сервером перед началом передачи данных. Он состоит из трёх этапов:
1. Клиент, который намеревается установить соединение, посылает серверу сегмент с порядковым номером (SN) и флагом SYN.
2. Сервер отвечает сегментом с флагами SYN и ACK. В поле номер подтверждения (ACK SN) указывается порядковый номер (SN) + 1.
3. Клиент отправляет сегмент с флагом ACK, подтверждая получение SYN-ACK от сервера.

### Обработка потерянных сегментов и управление передачей в TCP

TCP для обнаружения и обработки потерянных сегментов использует: повторную передачу по таймеру, квитирование и нумерацию сегментов. Для эффективного квитирования используется алгоритм скользящего окна.

#### Скользящее окно (Sliding window)

Особенность этого алгоритма состоит в том, что окно определено на множестве нумерованных байт неструктурированного потока данных, поступающих с верхнего уровня и буферизуемых протоколом TCP.Квитанция посылается в случае правильного приема данных. Отсутствие квитанции означает либо прием искаженного сегмента, либо потерю сегмента, либо потерю квитанции. В качестве квитанции получатель сегмента отсылает ответное сообщение (сегмент), в которое помещает число, на единицу превышающее максимальный номер байта в полученном сегменте.

#### Механизм повторной передачи (Retransmission)

В протоколе TCP главным средством, позволяющим определить необходимость повторной передачи пакета, является таймер, который отвечает за отсчет времени ожидания до повторной передачи (RTO, или Retransmission Timeout). Этот таймер запускается всякий раз, когда мы отправляем какой‑либо ТСР пакет и останавливается, когда получен пакет АСК. Когда пакет посылается, а получатель не отправляет в ответ подтверждающий пакет АСК, отправитель предполагает, что исходный пакет был потерян, и передает его еще раз. 

#### Контроль перегрузки (Congestion Control)

В TCP используется алгоритм аддитивного увеличения/мультипликативного уменьшения (**AIMD**) сочетает в себе линейный рост окна перегрузки с экспоненциальным уменьшением при возникновении перегрузки.

**Окно перегрузки** (CWND) определяет количество байтов, которые могут быть отправлены без подтверждения. Окно перегрузки поддерживается отправителем и служит для предотвращения перегрузки канала связи между отправителем и получателем слишком большим трафиком.

**Медленный старт** является частью стратегии управления перегрузкой, используемой протоколом TCP в сочетании с другими алгоритмами для предотвращения отправки большего объёма данных. Медленный старт начинается с окна перегрузки, равного от 1 до 10 MSS (максимальный размер сегмента). Размер окна перегрузки увеличивается на 1 MSS при получении подтверждения (фактически удвоение за RTT). При достижении порога медленного старта (ssthresh) TCP переключается на алгоритм предотвращения перегрузки. В этом случае окно увеличивается не более чем на `MSS*MSS/CWND` за каждое подтверждение (1 MSS за RTT). При обнаружении потери TCP предполагает перегрузку и сокращает нагрузку на сеть. Конкретные меры зависят от используемого алгоритма.

**TCP Tahoe**. При потере - повторная передача, половина текущего окна перегрузки сохраняется как порог перегрузки (`ssthresh = CWND/2`), и медленный старт начинается заново.

**TCP Reno**. Выполняется быстрая повторная передача, половина текущего окна перегрузки сохраняется как порог перегрузки и в качестве нового значения окна перегрузки (`CWND = ssthresh = CWND/2`), т. е. медленный старт пропускается, переходя к фазе предотвращения перегрузки. Общий алгоритм в таком случае называется быстрое восстановление.

### Практическая работа с TCP в Linux

#### 1. Создание TCP-прослушивателя с помощью `ncat`

Используя утилиту `ncat`, создадим минимально работающий сервер, который прослушивает TCP-соединение:
```
$ ncat -l 8888 -k -c "printf 'HTTP/1.1 200 OK\n\nTCP-listener-1\n\n'"
```

Параметры:

- `-l` (listen) - прослушивание указанного порта.
- `-k` (keep-open) - принимать несколько подключений в режиме прослушивания.
- `-c` (command) - выполнить команды для обработки входящего соединения (в нашем случае printf).

В другом терминале запустим другой TCP-прослушиватель:
```
$ ncat -l 8887 -k -c "printf 'HTTP/1.1 200 OK\n\nTCP-listener-2\n\n'"
```

#### 2. Проверка открытых портов с помощью `netstat`

С помощью утилиты `netstat` посмотрим прослушиваемые порты:
```
$ netstat -tuln
Active Internet connections (only servers)
Proto Recv-Q Send-Q Local Address           Foreign Address         State
...
tcp        0      0 0.0.0.0:8887            0.0.0.0:*               LISTEN
tcp        0      0 0.0.0.0:8888            0.0.0.0:*               LISTEN
...
```

Различие между утилитами `ncat` и `netstat`:

- `ncat` - утилита для создания сетевых соединений.
- `netstat` - утилита для мониторинга сетевых соединений и статистики.

#### 3. Тестирование соединения с помощью утилиты `curl`

Протестируем сервер, отправив HTTP-запрос на порт с помощью `curl`:
```
$ curl http://localhost:8888
TCP-listener-1
```
В ответе нет *HTTP/1.1 200 OK*, потому что `curl` автоматически интерпретирует и обрабатывает HTTP-заголовки, отображая только тело ответа.

#### 4. Создание UDP-прослушивателя с помощью `ncat`

С помощью утилиты `ncat` создадим UDP-прослушиватель на открытом TCP порте:
```
$ ncat -l 8888 -k -u -c "printf 'UDP-listener-1\n\n'"
```
Параметр `-u` указывает на использование UDP вместо TCP.

Посмотрим состояние портов:
```
$ netstat -tuln | grep 8888
tcp        0      0 0.0.0.0:8888            0.0.0.0:*               LISTEN
udp        0      0 0.0.0.0:8888            0.0.0.0:*
```

Как мы видим в выводе `netstat`, на одном и том же порту 8888 одновременно работают TCP и UDP.

#### 5. Отправка UDP/TCP-пакета с помощью `ncat`

UDP:
```
$ ncat -u localhost 8888
hello UDP                   <- Запрос (от клиента)
UDP-listener-1              <- Ответ (от сервера)
```

TCP:
```
$ ncat localhost 8888
HTTP/1.1 200 OK             <- Ответ (от сервера)

TCP-listener-1
```
