# Лекция 15 - Сеть в Docker

## Введение в Docker Networking

Когда мы работаем с Docker, контейнеры редко существуют в полной изоляции. Почти всегда нам необходимо, чтобы контейнеры подключались и взаимодействовали между собой и с сетевыми службами, не относящимися к Docker.

По умолчанию каждый контейнер имеет встроенную сетевую поддержку и может устанавливать исходящие подключения. У контейнер нет  информации о том, к какой сети он подключен, он видит только сетевой интерфейс с IP-адресом, шлюзом, таблицей маршрутизации и службами DNS.

При первом запуске Docker Engine автоматически создает встроенную сеть с названием "**default bridge**" (мост по умолчанию). Когда вы запускаете контейнер без опции `--network`, он подключается именно к этой сети. Контейнеры в этом мосту могут получать доступ к внешним сетевым службам. Они используют встроенный механизм маскардинга, что означает, что если у хоста Docker есть доступ в Интернет, то для доступа контейнера в Интернет не требуется дополнительной настройки.

Контейнеры в сети по умолчанию могут общаться между собой по IP-адресам, но не по именам. Чтобы контейнеры могли общаться между собой по именам и для изоляции групп контейнеров, Docker позволяет создавать **пользовательские сети** (**user-defined networks**).

Пример создания сети с использованием сетевого драйвера bridge и запуска контейнера в этой сети:
```
$ docker network create -d bridge setochka123
$ docker run --network=setochka123 -it go-app:multi-stage
```
Посмотрим список сетей:
```
$ docker network ls
NETWORK ID     NAME          DRIVER    SCOPE
...
fd86b0e03df7   setochka123   bridge    local
```

Пример связывания контейнеров в одну сеть:
```
docker run --name flask-app1 --network setochka123 -d -P flask-app:1.1
docker run --name flask-app2 --network setochka123 -d -P flask-app:1.0
```

## Сетевые драйверы в Docker

Docker предоставляет гибкую систему сетевых драйверов, которые позволяют настроить взаимодействие контейнеров именно так, как этого требует ваше приложение. В Linux доступны следующие встроенные сетевые драйверы:

### 1. `bridge`

С точки зрения Docker, мостовая сеть (`bridge`) использует программный мост, который позволяет контейнерам, подключенным к одной и той же мостовой сети, обмениваться данными, обеспечивая при этом изоляцию от контейнеров, которые не подключены к этой мостовой сети.

Мостовая сеть создает для контейнеров изолированную виртуальную сеть с собственной IPv4 (и опционально IPv6) подсетью, где каждый контейнер получает уникальный IP-адрес. По умолчанию эта сеть разрешает неограниченное взаимодействие между контейнерами внутри одной сети и с хост-машиной, но блокирует любой доступ извне и из других сетей Docker. Для связи с внешним миром драйвер использует два механизма: маскарадинг, который скрывает все контейнеры за IP-адресом хоста для исходящего интернет-трафика, и проброс портов (`publish`), который позволяет открывать доступ к конкретным сервисам контейнера на определенных портах хоста для входящих подключений.

#### Разница между пользовательскими мостовыми сетями (`user-defined bridges`) и мостовыми сетями по умолчанию:

- **Пользовательские мосты предоставляют автоматическую DNS резолюцию между контейнерами**. Контейнеры автоматически обнаруживают друг друга по именам в пользовательских мостах, тогда как в мостах по умолчанию требуется указывать IP-адреса.
- **Пользовательские мосты предоставляют лучшую изоляцию**. Создается защищенное сетевое пространство только для связанных контейнеров приложения, в то время как мост по умолчанию объединяет все незаданные явно контейнеры.
- **Контейнеры можно подключать и отключать от пользовательских мостов на лету**. Контейнеры можно динамически подключать/отключать от сети без перезапуска, тогда как для моста по умолчанию требуется остановка и перезапуск контейнера.
- **Конфигурируемость**. Каждый пользовательский мост можно настроить индивидуально под нужды. Для настройки моста по умолчанию необходимо перезапускать Docker.

#### Подключение контейнера к пользовательскому мосту

Когда вы создаете новый контейнер, вы можете указать один или несколько флагов `--network`. В примере ниже контейнер `flask-app` подключается к мосту `setochka123`, также пробрасывается порт 5000 на порт 8080 на хосте Docker, чтобы извне получить доступ к этому порту.

```
$ docker run --name flask-app123 --network setochka123 --publish 8080:5000 flask-app:1.1
```

Кроме подключения контейнера к сети на этапе создания, можно контейнеры подключать к сети во время работы:
```
$ docker run -d -P --name flask-app321 flask-app:1.0
$ docker network connect setochka123 flask-app321
```

Проверим подключение контейнера к сети:
```
$ docker network inspect setochka123
...
"Containers": {
            "d563ef8dc154f9d808f8194a9eab77127ff65f28c32fc9e7187544f13b9b682e": {
                "Name": "flask-app321",
                "EndpointID": "32a740e5c1615677beb8bf168d109b903b3594c355feef1c9acbd1527cf4815e",
                "MacAddress": "12:04:56:4e:8c:de",
                "IPv4Address": "172.18.0.2/16",
                "IPv6Address": ""
            }
...
```

Отключим контейнер от пользовательского моста:
```
docker network disconnect setochka123 flask-app321
```

### 2. `host`

При использовании `host` драйвера контейнер не получает собственного изолированного сетевого пространства, а вместо этого напрямую использует сетевой стек хоста (сетевые интерфейсы, порты и IP-адрес). Так как контейнер не имеет собственного IP-адреса, сопоставление портов не вступает в силу, и флаги `-p`, `--publish`, `-P` и `--publish-all` игнорируются, выдавая предупреждение:
```
WARNING: Published ports are discarded when using host network mode
```

#### Использование сетевого драйвера хоста может быть полезно в случаях:

- Оптимизации производительности
- В случаях, когда приложению необходимо большое количество портов

#### Пример использования

```
$ docker run --name flask-app123 --net=host flask-app:1.1
```

#### Ограничения

1. **Нельзя привязаться к IP-адресам хоста**.
2. **Функция host network в Docker Desktop работает на уровне 4**. Сетевые протоколы, которые работают ниже TCP или UDP, не поддерживаются.
3. **Конфликт с Enhanced Container Isolation**.
4. **Только Linux-контейнеры**.

### 3. `none`

Если вы хотите полностью изолировать сетевой стек контейнера, вы можете использовать флаг `--network none` при запуске контейнера. Внутри контейнера создается только loopback-интерфейс.

### 4. `overlay`

Драйвер `overlay` создает распределенную сеть между несколькими Docker демонами хостами. Эта сеть работает поверх (overlay) сетей отдельных хостов, позволяя подключенным контейнерам безопасно взаимодействовать при включенном шифровании. Docker автоматически обрабатывает маршрутизацию пакетов между нужными Docker хостами и контейнерами-получателями.

Пример создания `overlay` сети:
```
$ docker network create -d overlay --attachable oversetochka123
```

Использование `overlay` требует активированного Swarm-режима. Оверлейные сети часто используются для создания соединения между сервисами Swarm, но также можно использовать их для подключения автономных контейнеров, работающих на разных хостах. При использовании автономных контейнеров по-прежнему требуется Swarm-режим. Опция `--attachable` позволяет подключаться к оверлейной сети как автономным контейнерам, так и службам Swarm. Без опции `--attachable` к сети могут подключаться только службы Swarm.

### 5. `ipvlan`

Драйвер `IPvlan` предоставляет полный контроль над IPv4 и IPv6 адресацией контейнеров. В отличие от традиционных подходов с использованием `bridge`, `IPvlan` напрямую связывает контейнеры с физическим интерфейсом хоста или его под-интерфейсами, что делает сетевую виртуализацию более легковесной и производительной.

Режимы работы:

- l2 - коммутация на уровне MAC-адресов
- l3 - маршрутизация на IP-уровне (без broadcast)
- l3s - расширенный l3 режим

Пример создания `ipvlan` сети:
```
$ docker network create -d ipvlan \
    --subnet=192.168.1.0/24 \
    --gateway=192.168.1.1 \
    -o ipvlan_mode=l2 \
    -o parent=eth0 db_net
```

Пример подключения:
```
$ docker run --net=db_net container123
```

#### Ограничения:

- В l3 режиме отключен broadcast и multicast
- Требуется ручная настройка маршрутов между разными подсетями

#### Преимущества:

- Повышение производительности вследствии отсутствия `bridge`
- Полный контроль над IP-адресами
- Поддержка промышленных стандартов VLAN

### 6. `macvlan`

Драйвер `Macvlan` позволяет назначать контейнерам собственные MAC-адреса, делая их равноправными участниками физической сети. Каждый контейнер получает прямой доступ к сетевому интерфейсу хоста, что необходимо для устаревших приложений или приложений, мониторящих сетевой трафик.

Пример создания `macvlan` сети:
```
$ docker network create -d macvlan \
    --subnet=172.16.86.0/24 \
    --gateway=172.16.86.1 \
    -o parent=eth0 pub_net
```

#### Ограничения:

- Сетевое оборудование должно поддерживать режим прослушивания
- Риск исчерпания IP-адресов и MAC-адресов в сети

Если приложение может работать с использованием `bridge` или `overlay`, то лучше использовать их, так как эти решения могут оказаться более эффективными в долгосрочной перспективе.

## DNS между контейнерами

Вспомним, что DNS в сети по умолчанию (`default bridge`) не работает, контейнеры могут общаться только по IP-адресам. В пользовательских сетях DNS работает автоматически, т. е. контейнеры могут общаться по именам. При создании контейнера Docker регистрирует его имя и IP-адрес в DNS. Когда один контейнер обращается к другому по имени, DNS возвращает правильный IP-адрес. Для внешних сайтов Docker перенаправляет запросы на DNS-серверы хоста.

## Docker Compose

### Что такое Docker Compose?

Docker Compose - это инструмент для определения и запуска многоконтейнерных приложений Docker. Используя Docker Compose, можно настроить сервисы, сети, тома и другие ресурсы вашего приложения в одном конфигурационном YAML-файле, а затем запустить всё одной командой.

### CLI и ключевые команды

Интерфейс CLI Docker позволяет вам взаимодействовать с вашими приложениями Docker Compose с помощью команды `docker compose` и ее подкоманд. Используя CLI, можно управлять жизненным циклом ваших многоконтейнерных приложений, определенных в compose.файл yaml.

#### Ключевые команды:

- Для запуска всех служб, определенных в файле `compose.yaml`:

    ```$ docker compose up```

- Для остановки и удаления всех запущенных служб:

    ```$ docker compose down```

- Для просмотра логов:

    ```$ docker compose logs```

- Для просмотра запущенных сервисов:

    ```$ docker compose ps```

Для более подробного ознакомления с этими и другими командами обращайтесь к [официальной документации](https://docs.docker.com/reference/cli/docker/compose/).

### Пример использования Docker Compose

Рассмотрим небольшой пример использования docker compose для развертывания приложения, состоящего из двух сервисов: веб-приложения python и базы данных redis.

Compose.yaml:
```
services:
  webapp:
    build: .                            # Сборка образа из текущей директории
    container_name: my-webapp
    ports:                              # Проброс портов
      - "5000:5000"
    environment:                        # Объявление переменной окружения
      - REDIS_URL=redis://redis:6379
    depends_on:                         # Перед запуском должен быть запущен redis
      - redis
    networks:                           # Подключение к пользовательской сети
      - my-app-network
    deploy:
      resources:
        limits:                         # Ограничение ресурсов
          memory: 256M
          cpus: "0.5"
        reservations:                   # Резервирование ресурсов
          memory: 128M
          cpus: "0.25"

  redis:
    image: redis:7-alpine
    container_name: my-redis
    networks:
      - my-app-network
    deploy:
      resources:
        limits:
          memory: 128M
          cpus: "0.3"

networks:                               # Пользовательская сеть
  my-app-network:
    driver: bridge
```

В этом файле декларативно описаны необходимые сервисы, их конфигурации и взаимосвязи. Compose автоматизирует развертывание всего приложения: собирает образы, создает сети, ограничивает ресурсы контейнеров, настраивает переменные окружения и пробрасывает порты для доступа к приложению извне.

Dockerfile для сборки веб-приложения:
```
FROM python:3.9-alpine

WORKDIR /app

RUN pip install flask redis

COPY ./app/app.py .

CMD ["python", "app.py"]
```

Для запуска приложения необходимо ввести одну команду (терминал должен быть открыт в директории с yaml-файлом):
```
$ docker compose up
```

Просмотрим запущенные сервисы:
```
$ docker compose ps
NAME        IMAGE                 COMMAND                  SERVICE   CREATED          STATUS          PORTS
my-redis    redis:7-alpine        "docker-entrypoint.s…"   redis     13 seconds ago   Up 13 seconds   6379/tcp
my-webapp   compose-test-webapp   "python app.py"          webapp    13 seconds ago   Up 13 seconds   0.0.0.0:5000->5000/tcp
```

Остановим приложение, удалив контейнеры и сеть:
```
$ docker compose down
```
