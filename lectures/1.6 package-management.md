# Лекция 6 - Управление пакетами

В стародавние времена, на заре появления Linux, установка софта на данную ОС была весьма нетривиальной задачей. Для начала, надо было найти исходный код программы, все необходимые зависимости, которые зачастую конфликтовали с теми, что уже установлены в системе. Также пользователь сталкивался с неприятной настройкой конфигов с нуля, а в худшем случае "make install" мог стать последним гвоздем в крышку гроба работоспособности ОС.

Поэтому все современные дистрибутивы Linux стремятся к наилучшему на данный момент способу решения этих проблем: управление пакетами. Эту задачу берет на себя программа, которая так и называется - **пакетный менеджер**. Он отвечает за процессы установки, обновления и удаления пользовательского ПО и компонентов системы. 

Сам софт распространяется в виде **пакетов**, которые могут содержать в себе от ссылок на исходный код и инструкций по сборке до готовых к запуску бинарников, упакованных рядом с требуемыми версиями библиотек. Второй приведенный вариант, очевидно, хуже, хоть и весьма распространен среди проприетарного (т.е без открытого исходного кода)"профессионального" софта, вроде DAW, программ для монтажа видео и т.д.

Существует бесчисленное множество пакетных менеджерах, мы остановимся на перечислении основных:

- **dpkg**. Имеет формат пакета .deb, впервые появился и используется в ОС, происходящих от дистрибутива Debian. Довольно старый, имеет множество ограничений. Тем не менее, представляет крайне удобный формат пакета. По скорости работы ничем не выделяется, хорошо, что тормозит крайне редко. Обычно оборачивается оболочкой apt, которую можно увидеть в современных дистрибутивах типа Ubuntu.
- **rpm**. Пакетный менеджер в ОС на основе RedHat Enterprise Linux. Он обычно оборачивается в dnf, который довольно производительно разрешает зависимости пакетов.
- **pacman**. ПМ, созданный специально для Arch Linux. Является крайне производительным, на практике устанавливает приложение быстрее любой другой альтернативы. Автор лекций использует Fedora Linux, который нативно поддерживает только rpm, но в виду скокрости работы pacman, устанавливает его отдельно для последующей работы с пакетами.

# Препарируя Пакет

Рассмотрим из чего состоит пакет. Большинство пакетов, как указано выше, содержат скомпилированное приложение и метаинформацию, которая позволяет пакетному менеджеру работать с конкретной версией этого ПО.

Файл метаинформации имеет свой формат, у всех типов пакетов разный. Он содержит в себе имя, номера версии и сборки, сведения о разработчике, и, главное, зависимости. Отдельно стоит упомянуть установочные и настроечные сценарии, представляющие собой скрипты для корректной установки и первоначальной настройки приложения соответственно. Не забудем и пост-инсталяционную конфигурацию. В ней можно наблюдать разного рода очистки после установки. Как многие могли догадаться, для удаления ПО пакет также содержит отдельный сценарий.

# Конфликты

Нередко случается ситуация, когда при установке пакета, возникает конфликт с уже установленным пакетом. Из наиболее распространенных причин выделим:

- Оба пакета включают файл с одинаковым путем
- Оба пакета представляю собой сетевую службу, которая работает на одном и том же порту
- Это тот же самый пакет из другого дистрибутива

Разные пакетные менеджеры реагируют на эту ситуацию по-разному: dpkg просто окажется от дальнейшей установки, apt предложит удалить пакет, мешающий установке нового (у меня так товарищ "снес" жизненно важный пакет и потом целый день занимался восстановлением системы).

# Репозиторий

Репозиторий пакетного менеджера — это удаленное или локальное хранилище программных пакетов, которое выступает в качестве источника для установки, обновления или удаления программного обеспечения на компьютере через менеджер пакетов. Эти хранилища, часто расположенные на серверах, содержат не только сами пакеты, но и метаданные о них, а также могут быть разделены на официальные (поддерживаются разработчиками ОС), коммерческие и неофициальные (поддерживаются сообществом).

Отечественные производители ОС отличились: вы можете приобрести диск с актуальным (на какой-то момент) репозиторием и производить установку и обновление программ с него, без подключения к Интернету.

# Основные операции (на примере APT)

Обновление списка пакетов:
```
# apt update
```

Установка пакета из репозитория:
```
# apt install firefox
```

Удаление установленног пакета:
```
# apt remove firefox
```

Удаление пакета вместе со всеми конфигами:
```
# apt remove --purge firefox
```

# Недостатки управления пакетами
Наконец, обсудим неостатки такого подхода. Существует мнение, что это плохая инженерная идея: проблемы при установке пакета из за конфликтующих зависимостей в системе, вложенные зависимости, которые иногда приводят к длительной установке софта. Яркий тому пример - пакетный менеджер homebrew на MacOS. В нем часто можно столкнуться с ситуацией, когда вложенные зависимости начинают собираться с нуля, так как автор пакета не указал зависимость в бинарном виде.

Однако на данный момент не существует более удачного способа решать проблемы зависимостей и конфликтов. Поэтому индустриальный стандарт - работа с пакетами!
