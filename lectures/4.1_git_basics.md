# Лекция 16 - Основы git

Git - это распределенная система контроля версий (Version Control System, VCS).

Git решает следующие задачи:

- Хранение нескольких версий кода. Git позволяет создавать изолированные "ветки" для разработки новых функций или исправления ошибок.
- Откат к прошлым версиям. Git сохраняет коммиты проекта на всех этапах разработки, что позволяет вернуться к предыдущим версиям.
- Совместная работа и идентификация автора кода. Git позволяет нескольким разработчикам работать над одним проектом параллельно, не мешая друг другу.

## Создание репозитория

Опустим этап установки и настройки git. Если с этим моментом возникают трудности рекомендуем обратиться к [официальному источнику](https://git-scm.com/).

Репозиторий - это хранилище проекта и его истории.

Для создания Git-репозитория в текущей директории необходимо использовать команду `git init`, которая инициализирует скрытую директорию `.git`, в которой хранится служебная информация.

Рассмотрим следующую последовательность команд:
```
$ mkdir git_test
$ cd git_test
$ ls -la
$ git init
$ ls -la
```
Пояснение:  
При создании директории `git_test` в ней изначально пусто, в чём мы убеждаемся, применив команду `ls -la`. После инициализации git-репозитория (применение команды `git init`) в рабочей директории появится новая скрытая директория, что мы проверяем, снова вызвав команду `ls -la`. Результат вызова последней команды:

```
... ./
... ../
... .git/
```

Действительно появилась скрытая директория `.git/`.

## Коммит

Коммит (commit) - это фиксация текущих изменений, внесённых в репозиторий, и сохранение их в истории. Выполнить такую фиксацию можно с помощью команды `git commit`.

Рассмотрим следующие команды и их выводы (предварительно добавим файл в наш созданный репозиторий):
```
$ git status
On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Mat_Model.py

nothing added to commit but untracked files present (use "git add" to track)

$ git add Mat_Model.py

$ git status
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   Mat_Model.py

$ git commit
[master (root-commit) 674ed7c] add Mat_model
 1 file changed, 436 insertions(+)
 create mode 100644 Mat_Model.py

$ git status
On branch master
nothing to commit, working tree clean

$ git log --oneline
674ed7c (HEAD -> master) add Mat_model
```

Пояснение:

- Команда `git status` отображает состояние рабочего каталога и индекса. Добавив файл в репозиторий и использовав команду `git status`, мы видим, что у нас есть файл со статусом untracked.
- Команда `git add` добавляет содержимое файлов в индекс (добавляет файлы в коммит). После добавления файла в индекс, он теперь подготовлен к коммиту, что мы видим, повторно вызвав команду `git status`.
- Вызвав команду `git commit`, откроется редактор для того, чтобы написать сообщение к коммиту. Вместо использования редактора можно использовать флаг `-m` с дальнейшим указанием сообщения (пример: `$ git commit -m "add Mat_Model"`). В выводе Git предоставил обратную связь по коммиту.
- Команда `git log` выводит историю изменений в репозитории, т. е. выводит историю коммитов. Использование флага `oneline` укорачивает вывод команды.

Для просмотра изменения в файлах до коммита можно использовать комманду `git diff` (файлы в состоянии modified). Использование файла `--staged` покажет разницу между staged-файлами и последним коммитом.

Рассмотрим пример изменения кода в файлу Mat_Model.py:
```
$ git diff
diff --git a/Mat_Model.py b/Mat_Model.py
index 1e9b18d..da92b8c 100644
--- a/Mat_Model.py
+++ b/Mat_Model.py
...
-DEFAULT_TIMER_FEED = 30                 # T1
-DEFAULT_TIMER_MELT = 5                  # T2
-DEFAULT_TIMER_HEAT = 60                 # T3
+DEFAULT_TIMER_FEED = 15                 # T1
+DEFAULT_TIMER_MELT = 20                 # T2
+DEFAULT_TIMER_HEAT = 70                 # T3
...

$ git diff --staged

$ git add Mat_Model.py

$ git diff

$ git diff --staged
diff --git a/Mat_Model.py b/Mat_Model.py
index 1e9b18d..da92b8c 100644
--- a/Mat_Model.py
+++ b/Mat_Model.py
...
-DEFAULT_TIMER_FEED = 30                 # T1
-DEFAULT_TIMER_MELT = 5                  # T2
-DEFAULT_TIMER_HEAT = 60                 # T3
+DEFAULT_TIMER_FEED = 15                 # T1
+DEFAULT_TIMER_MELT = 20                 # T2
+DEFAULT_TIMER_HEAT = 70                 # T3
...
```

Пояснение:

Пока измененный файл не был добавлен в индекс, его изменения отображаются командой `git diff`, после добавления файла в индекс его изменения отображаются командой `git diff --staged`.

## Индекс (Staging area)

Индекс (он же область подготовки или staging area) - это промежуточное простраство между рабочим каталогом и репозиторием, в котором формируется содержание следующего коммита.

Основными командами для работы с индексом являются: `git add`, `git status`, `git commit` и `git reset`. С первыми тремя командами мы уже знакомы. Последняя (`git reset`) отменяет все изменения сделанные после последнего коммита.

## Состояния файлов в Git

При  добавлении файла в директорию репозитория он является неотслеживаемым (**untracked**), т. е. git не следит за изменениями в нём. После применения команды `git add` к данному файлу он переходит в состояние **staged**. Данное состояние указывает на то, что файл проиндексирован, т. е. отмечен для включения в следующий коммит. После применения команды `git commit` файл переходит из состояния **staged** в **unmodified**, которое указывает, что файл не содержит изменений по сравнению с последней зафиксированной версией. После внесения изменений в файл он переходит в состояние **modified**, что указывает на то, что файл был изменён и не проиндексирован.

Файлы могут быть отслеживаемыми и неотслеживаемыми, а отслеживаемые файлы, в свою очередь, могут принимать следующие состояния: 
- Unmodified.
- Modified.
- Staged.

Подробнее с состояниями файлов можете ознакомиться в [официальной документации](https://git-scm.com/book/ru/v2/Основы-Git-Запись-изменений-в-репозиторий), в которой есть визуализальное представление переходов между состояниями.

## Игнорирование файлов

Файл `.gitignore` - текстовый файл, который указывает Git, какие файлы или папки следует игнорировать и не добавлять в репозиторий.

В наш уже имеющийся репозиторий добавим файл `text.txt`, который мы не хотим добавлять в репозиторий. Для этого создаём файл `.gitignore`, в котором указывает файл `text.txt`. Содержание `.gitignore`:
```
text.txt
```

Проверим статус индекса:
```
$ git status 
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        text.txt

nothing added to commit but untracked files present (use "git add" to track)

$ nano .gitignore

$ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore

nothing added to commit but untracked files present (use "git add" to track)
```

Пояснение:

При первом вызове команды `git status` у нас появился новый файл `text.txt`, который после мы добавили в `.gitignore`. Соответственно, при втором вызове `git status` файл `text.txt` игнорируется.

## Ветвление в Git

Ветка (branch) - это перемещаемый указатель на один из коммитов. Ветки позволяют производить изолированную параллельную работу над одним проектом.

### Создание веток и переключение между ними

Для создания ветки можно использовать команду `git branch` с названием ветки:
```
$ git branch test_branch
```
Для просмотра списка всех веток также используется команда `git branch`:
```
$ git branch
* master
  test_branch
```
Ветка `master` - это основная ветка, а `test_branch` - недавно созданная. Звёздочка указывает, в какой ветке мы сейчас находимся.

Для переключения между ветками используется команда `git checkout` с именем ветки:
```
$ git checkout test_branch
Switched to branch 'test_branch'

$ git branch
  master
* test_branch
```

Можно использовать команду `git checkout -b` с именем для создания ветки и мнгновенного переключения на неё:
```
$ git branch
  master
* test_branch

$ git checkout -b test_branch1
Switched to a new branch 'test_branch1'

$ git branch
  master
  test_branch
* test_branch1
```

### Слияние веток

Допустим, при работе с проектом на одной из веток мы хотим добавить новый файл и закоммитить его. Для этого выполним следующие команды:
```
$ git status
On branch test_branch1
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        text1.txt

nothing added to commit but untracked files present (use "git add" to track)

$ git add text1.txt

$ git commit -m "add text1"
[test_branch1 24a1f42] add text1
 1 file changed, 1 insertion(+)
 create mode 100644 text1.txt
```

При переходе на ветку master мы обнаружим, что файл `text1.txt` исчезнет. Далее мы хотим произвести те же изменения на ветке master, для этого используем команду `git merge test_branch1`:
```
$ git merge test_branch1
Updating 50a4f38..24a1f42
Fast-forward
 text1.txt  | 1 +
 1 files changed, 1 insertions(+)
 create mode 100644 text1.txt
```

Изменения в ветках также можно проследить, используя команду `git log`. Для слияния можно использовать `git rebase`. Хоть эти две команды и схожи, они имеют разный принцип работы: `git merge` объединяет ветки, сохраняя историю изменения обеих веток, `git rebase` перемещает все коммиты одной ветки и ставит их в конец другой.

### Конфликт слияния

Конфликт слияния возникает, когда Git не может автоматически объединить изменения из двух веток, например, потому что они редактировали одни и те же строки в одном и том же файле.

Допустим, мы перешли на ветку `test_branch1`, изменили содержание файла `text1.txt`, закоммитили изменения, вернулись на ветку мастер и сделали то же самое, но внесли другие изменения в файл `text1.txt`. В таком случае при автоматическом слиянии возникнет конфликт:
```
$ git merge test_branch1
Auto-merging text1.txt
CONFLICT (content): Merge conflict in text1.txt
Automatic merge failed; fix conflicts and then commit the result.
```

При открытии файла мы увидим, что он содержит изменения с обеих веток. Отредактируем файл и закоммитим его:
```
$ git add text1.txt

$ git commit
```

### cherry-pick

`git cherry-pick` - команда, которая позволяет взять конкретный коммит из одной ветки и применить его изменения в текущей ветке.

Допустим мы внесли несколько коммитов в ветке `test_branch1`, но применить хотим лишь один. Для этого введем следующие команды:
```
$ git log test_branch1
commit 77540665913262e293036a0eb1b983c3b31f7cef (test_branch1)
...

    add text3.txt

commit 2af8ae093febc05201711c8b91444e5e0f65d460
...

    add text2.txt

...

$ git cherry-pick 77540665913262e293036a0eb1b983c3b31f7cef #хэш последнего коммита
[master 353f09a] add text3.txt
 1 file changed, 1 insertion(+)
 create mode 100644 text3.txt

$ ls
Mat_Model.py  text.txt  text1.txt  text3.txt
```

Таким образом, игнорируется коммит с добавлением `text2.txt` и добавляется только файл `text3.txt`.
