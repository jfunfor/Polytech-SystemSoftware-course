# Лекция 12 - Основы контейнеризации и архитектура Docker

## Что такое контейнеризация и чем они отличается от виртуализации? 
Представьте, что вы разрабатываете крутое веб-приложение, которое состоит из трех основных компонентов — фронтенда на React, API на Python и базы данных PostgreSQL. Если бы вы захотели поработать над этим проектом, вам пришлось бы установить Node, Python и PostgreSQL.

Как убедиться, что у вас те же версии, что и у других разработчиков в вашей команде?
Как гарантировать, что версия Python (или Node, или базы данных), которая нужна вашему приложению, не зависит от того, что уже установлено на вашей машине? Как справиться с потенциальными конфликтами?

В этом нас и выручает контейнеризация!

Что такое контейнер? Если просто, **контейнеры** — это изолированные процессы для каждого из компонентов вашего приложения. Каждый компонент — фронтенд-приложение на React, движок API на Python и база данных — работает в своей собственной среде, полностью изолированной от всего остального на вашей машине.

Вот что делает эту тему такой привлекательной. Контейнеры:

* **Самостоятельные**. Каждый контейнер имеет всё необходимое для функционирования и не зависит от каких-либо предустановленных зависимостей на хост-машине.
* **Изолированные**. Поскольку контейнеры запускаются в изоляции, они минимально влияют на хост-машину и другие контейнеры, что повышает безопасность ваших приложений.
* **Независимые**. Каждым контейнером управляют независимо. Удаление одного контейнера не повлияет на другие.
* **Портативные**. Контейнеры могут работать где угодно! Контейнер, который работает на вашей машине для разработки, будет так же работать в дата-центре или в облаке!

Что касается отличий контейнеризации и вирутализации то, не вдаваясь в детали, виртуальная машина — это полноценная операционная система со своим собственным ядром, драйверами оборудования, программами и приложениями.
Контейнер — это просто изолированный процесс со всеми файлами, которые ему нужны для работы. 

## Какие преимущества дают контейнеры по сравнению с виртуальными машинами?
Запускать виртуальную машину только для изоляции одного приложения — это большие издержки. В то же время, если вы запускаете несколько контейнеров, все они используют одно и то же ядро, что позволяет запускать больше приложений на меньшей инфраструктуре.  

Переносимость и легковесность контейнеров также позволяют легко управлять рабочими нагрузками в динамическом режиме, масштабируя или сокращая приложения и сервисы по мере потребностей бизнеса в реальном времени. 

Подводя итог, контейнеризация предоставляет собой жизнеспособную и экономически эффективную альтернативу виртуальным машинам, что позволяет использовать больше ресурсов вашего сервера для достижения целей. Такой метод идеально подходит для сред с высокой плотностью размещения, а также для небольших и средних развертываний, где нужно делать больше с меньшими ресурсами.

## Что такое Docker?
**Docker** — это открытая платформа для разработки, поставки и запуска приложений. Docker позволяет вам отделить ваши приложения от вашей инфраструктуры, чтобы вы могли быстро поставлять программное обеспечение. С помощью Docker вы можете управлять своей инфраструктурой теми же способами, что и своими приложениями. Используя преимущества методологий Docker для поставки, тестирования и развертывания кода, вы можете значительно сократить задержку между написанием кода и его запуском в рабочей среде.

Docker предоставляет инструменты и платформу для управления жизненным циклом ваших контейнеров:

* Вы можете разрабатывать свое приложение и его вспомогательные компоненты с использованием контейнеров.

* Контейнер становится единицей распределения и тестирования вашего приложения.

* Когда вы готовы, вы можете разворачивать свое приложение в требуемой среде в виде контейнера или оркестрируемого сервиса. Это работает одинаково независимо от того, является ли ваша производственная среда локальным дата-центром, облачным провайдером или гибридом из двух.

## Из каких компонентов состоит Docker?
**Архитектура Docker:** 

![](https://docs.docker.com/get-started/images/docker-architecture.webp)

Docker использует клиент-серверную архитектуру. Docker-клиент взаимодействует с Docker daemon, который выполняет тяжелую работу по сборке, запуску и распределению ваших Docker-контейнеров. Docker-клиент и daemon могут работать в одной системе, или можно подключить Docker-клиент к удаленному Docker daemon. Docker-клиент и daemon общаются с помощью [REST API](https://ru.wikipedia.org/wiki/REST) через UNIX-сокеты или сетевой интерфейс. Еще одним Docker-клиентом является Docker Compose, который позволяет работать с приложениями, состоящими из набора контейнеров.

### Docker daemon

Docker daemon (`dockerd`) ожидает запросы от Docker API и управляет объектами Docker, такими как образы, контейнеры, сети и тома. Daemon также может взаимодействовать с другими daemons для управления службами Docker.

### Docker client

Docker-клиент (`docker`) — это основной способ, с помощью которого многие пользователи взаимодействуют с Docker. Когда вы используете команды, такие как `docker run`, клиент отправляет эти команды daemon `dockerd`, который их выполняет. Docker-клиент может взаимодействовать более чем с одним daemon.

### Docker registries

Реестр Docker хранит образы Docker. Docker Hub — это публичный реестр, который может использовать каждый, и Docker по умолчанию ищет образы в Docker Hub. Вы даже можете запустить свой собственный частный реестр.

Когда вы используете команды `docker pull` или `docker run`, Docker извлекает необходимые образы из вашего настроенного реестра. Когда вы используете команду `docker push`, Docker отправляет ваш образ в настроенный реестр.

## Что такое образ и Dockerfile? 
**Образ** — это неизменяемый шаблон с инструкциями по созданию контейнера Docker. Часто образ основан на другом образе с некоторыми дополнительными настройками. Например, вы можете собрать образ на основе образа `ubuntu`, но с установленным веб-сервером Apache и вашим приложением, а также с деталями конфигурации, необходимыми для запуска вашего приложения.

Вы можете создавать свои собственные образы или использовать только образы, созданные другими и опубликованные в реестре. Чтобы собрать свой собственный образ, вы создаете Dockerfile с простым синтаксисом для определения шагов, необходимых для создания образа и его запуска. Каждая инструкция в Dockerfile создает слой в образе. Когда вы изменяете Dockerfile и пересобираете образ, перестраиваются только те слои, которые были изменены. Это одна из причин, почему образы такие легковесные, маленькие и быстрые по сравнению с другими технологиями виртуализации.

## Из чего состоит образ? Как работает файловая система контейнера (Union File System)?

Из сказанного выше уже ясно, что образы контейнеров состоят из слоёв. И каждый из этих слоёв, после создания, является неизменяемым. Но что это на самом деле означает? И как эти слои используются для создания файловой системы, которую может использовать контейнер? Рассмотрим ниже 

### Слои образа
Каждый слой в образе содержит набор изменений файловой системы — добавления, удаления или модификации. Давайте рассмотрим теоретический образ:

![](https://docs.docker.com/get-started/docker-concepts/building-images/images/container_image_layers.webp)

1. Первый слой добавляет базовые команды и менеджер пакетов, такой как apt.

2. Второй слой устанавливает среду выполнения Python и pip для управления зависимостями.

3. Третий слой копирует определенный файл с требованиями приложения - requirements.txt.

4. Четвертый слой устанавливает конкретные зависимости приложения из этого файла.

5. Пятый слой копирует непосредственно исходный код приложения.

Это выгодный подход, потому что позволяет повторно использовать слои между образами. Например, представьте, что вы хотите создать другое приложение на Python. Благодаря послойной структуре вы можете использовать ту же самую базовую среду Python. Это ускорит сборку и сократит объем памяти и пропускной способности, необходимых для распространения образов. Наложение образов может выглядеть следующим образом:

![](https://docs.docker.com/get-started/docker-concepts/building-images/images/container_image_layer_reuse.webp)

Слои позволяют вам расширять образы других пользователей, повторно используя их базовые слои, что позволяет добавлять только те данные, которые нужны вашему приложению.

### Наложение слоев
Наложение слоев стало возможным благодаря content-addressable storage и union filesystems.

1. После загрузки каждого слоя он извлекается в свою собственную директорию в файловой системе хоста.
2. Когда вы запускаете контейнер из образа, создается union filesystem, где слои накладываются друг на друга, создавая новое унифицированное представление.
3. При запуске контейнера его root директория устанавливается в расположение унифицированной директории (union filesystem) c помощью `chroot`.

При создании union filesystem, помимо слоев образа, создается директория специально для работающего контейнера. Это позволяет контейнеру вносить изменения в файловую систему, в то время как исходные слои образа остаются нетронутыми. Это позволяет вам запускать несколько контейнеров из одного базового образа.

## Как Docker использует пространства имён (namespaces) и контрольные группы (cgroups)?

### namespaces
Docker написан на языке программирования Go и использует несколько функций ядра Linux для обеспечения своей функциональности. Docker использует технологию под навзанием `namespaces` для предоставления изолированного рабочего пространства, которое называется контейнером. Когда вы запускаете контейнер, Docker создаёт набор пространств имён для этого контейнера.

Эти пространства имён обеспечивают слой изоляции. Каждый аспект контейнера работает в отдельном пространстве имён, и его доступ ограничен этим пространством имён.

### [Control Groups](https://www.kernel.org/doc/Documentation/cgroup-v1/cgroups.txt) 
Группы управления (cgroups) являются компонентом Linux-контейнеров. Они реализуют учёт и ограничение ресурсов. Они предоставляют множество полезных метрик, а также контролируют распределение ресурсов, чтобы каждый контейнер получал свою справедливую долю памяти, процессорного времени, и, что более важно, чтобы один контейнер не мог вывести систему из строя, исчерпав один из этих ресурсов.

Таким образом, хоть группы управления не играют роли в предотвращении доступа одного контейнера к данным и процессам другого контейнера или воздействия на них, они необходимы для отражения некоторых атак типа «отказ в обслуживании». Они особенно важны в мультитенантных платформах (где один экземпляр приложения обслуживает множество клиентов), таких как публичные и частные PaaS, чтобы гарантировать стабильное время бесперебойной работы (и производительность), даже когда некоторые приложения начинают работать некорректно.

## Что такое DockerHub и для чего он нужен? Какие ещё container registry бывают?

**Docker Hub**

> Docker Hub - крупнейший в мире реестр контейнеров для хранения, управления и обмена образами Docker. Он бесшовно интегрируется с вашими инструментами, повышает производительность и обеспечивает надежное развертывание, распространение и доступ к контейнеризированным приложениям. 

Docker Hub также предоставляет разработчикам предварительно собранные образы и ресурсы для ускорения рабочих процессов разработки.

Ключевые особенности:

* Неограниченное количество публичных репозиториев

* Приватные репозитории

* Webhooks для автоматизации рабочих процессов

* Интеграции с GitHub и Bitbucket

* Параллельные и автоматизированные сборки

* Проверенный контент, включающий высококачественные и безопасные образы

Помимо Docker Hub, существует несколько других популярных container registry, каждый из которых предлагает уникальные преимущества и особенности.

**GitHub Container Registry (GHCR)**

> [GHCR](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry#about-the-container-registry) - реестр от GitHub, тесно интегрированный с платформой. Он позволяет хранить образы контейнеров как в рамках вашей личной учетной записи, так и в организациях, с возможностью привязки образов к конкретным репозиториям. 

Одной из ключевых особенностей GHCR является гибкая система управления доступом: вы можете наследовать разрешения непосредственно от связанного репозитория либо настроить детализированные права доступа независимо от репозитория, что обеспечивает точный контроль над распространением образов.

GHCR тесно интегрирован с GitHub Actions, позволяя автоматизировать процессы сборки и публикации контейнерных образов непосредственно из репозитория.

**Azure Container Registry (ACR)**

> [ACR](https://azure.microsoft.com/en-us/products/container-registry) - реестр контейнеров от Microsoft, глубоко интегрированный в экосистему Azure. Он поддерживает георепликацию, позволяя обслуживать пользователей из разных регионов через единый реестр, и предлагает автоматизированную сборку образов через задачи ACR. 

Реестр поддерживает не только образы Docker, но и Helm-чарты и другие артефакты OCI (Open Container Initiative), предоставляя единое решение для работы с контейнерами в среде Azure.

ACR тесно взаимодействует с Azure Kubernetes Service (AKS) и Azure Machine Learning, а также предоставляет расширенные функции безопасности благодаря встроенной интеграции с виртуальной сетью Azure и Microsoft Entra ID.
